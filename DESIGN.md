# Logos: A Unified Programming System

*The consolidated design document for Logos: vision and architecture first, representation and subsystem detail below. It reads top-down by altitude. Some later sections (identity recognition, representation internals) describe target architecture not yet validated against a real corpus; those are flagged inline.*

## The core vision

Logos's foundational commitment is **radical unification**: every piece of logic the system contains — programs, logos, proofs, compilation rules, optimization passes, the compiler itself, documentation, tooling, and the language's own parsing rules — lives in a single data structure, the **Logic Graph**. There is no separation between "the language" and "what is written in it." Everything reachable in the system is queryable and manipulable through the same meta-reflection.

The bet is that the usual boundaries — language vs. compiler, code vs. specification, program vs. proof, source vs. tooling — are accidents of how systems were historically built, not necessities. Collapsing them yields a system that is simpler at its core, more expressive in what it can state, and more honest about what it is.

Logos is also a serious systems language: static memory safety via a borrow checker (lexical lifetimes, no GC, explicit ownership and moves), zero-cost abstraction, capability tracking for effects, and a layered architecture compiling to multiple backends. The unification does not cost much performance. Logic Graph code is interpreted by default; an immutable (frozen) region can be explicitly JIT compiled with Cranelift, which is faster in most cases, whereas mutable code stays interpreted — compiling it would only be invalidated the moment its structure changed. Reflection needs an LG form to exist, not a compiled one — it cannot be done on anything lower than the LG — so JITed code stays fully reflectable through the LG it was compiled from, and interpreted code simply *is* its LG. The seed is the one running thing that is neither interpreted nor JITed: it is compiled Rust that behaves like frozen, JITed Logos, never mutated at runtime (it is all compiled, and one would not want to borrow every seed identity). Its reflectability is a target reached incrementally, not a v1 property: v1 ships the seed's primitives as `native` — callable-only machine code, opaque to reflection — and each becomes reflectable as LG only once it is ported to Logos source, so the seed's logic reflects in full only at the self-hosted end state (see *The architecture*).

## Why now: a substrate for machine-written code

The unification thesis is old; what makes it urgent now is who writes the code. A fully reflective, rewritable structure that carries its own logos and proofs was, for most of computing history, a luxury: humans wrote the code, and humans mostly did not need the machine to reason about the code's own structure. That is part of why the closest ancestors stayed niche. When models write most of the code, that same property flips from luxury to requirement.

The reason is the feedback loop. A model that edits *text* pushes a guess through a fragile toolchain and hopes it holds. A model that edits a Logic Graph rewrites a structure that already carries its scopes, logos, borrow states, and (above the seed) proofs, and gets machine-checked feedback that the change is correct and safe before it runs. The same reader-writer rule that governs memory also governs self-modifying code (see *Representation, elaboration, and lifetimes*), so a program can improve its own code iteratively without being permitted to break it. Machine-checked correctness is the direction this is built toward, because that is what a non-human author most needs and least reliably supplies on its own.

This is precisely the gap in the two systems Logos learns from, each of which holds one half and not the other:

- **Smalltalk** is the malleable half: a live, fully reflective image that can inspect and rewrite itself at runtime. What it lacks is any correctness floor. It is dynamically typed, with no borrow checker, no static guarantees, and no way to prove a self-modification is right. Its reflection is runtime-only and carries no types, lifetimes, or proofs, and it is image-based and garbage-collected rather than a bare-metal systems substrate. Malleable, but unsafe.
- **Lean** is the proven half: machine-checked proofs down to a small trusted kernel, self-hosted, with real metaprogramming. What it lacks is being a general systems substrate. It is garbage-collected and functional, not built for systems, GPU, or native-speed performance; it keeps surface syntax, elaboration, and kernel term language as separate layers rather than one uniform structure the running program rewrites; and its verification is aimed at mathematicians proving theorems, not at a graded floor that scales from a borrow check to a full proof over the same code. Proven, but not a substrate a program rewrites and runs fast.

Neither has both halves. Logos aims for the union on a systems base: rewrite it as freely as Smalltalk, check it as strictly as Lean, run it as fast as Rust, all in one structure. The other sections describe how (the single reader-writer rule, the graded verification strata, interpret-by-default with opt-in compilation); this section is why that particular combination is worth the size, and why now.

## The architecture

**The Logic Graph is the primary representation.** Not an AST (syntax-only) and not a conventional IR (post-elaboration codegen form), it is the unified structure holding the program with all semantic information attached (resolved scopes, inferred logos, borrow states, propagated capabilities), the rules that governed its parsing and elaboration, the standard library, the compiler's own logic, and eventually proofs about all of it. Navigation is uniform: the same operations users employ on their own code traverse any subgraph, including the compiler's.

**A small Rust bootstrap seed starts the system.** The seed is the minimum: a parser producing Logic Graph nodes, an evaluator for those nodes, enough logos machinery to check the kernel, and a path to Cranelift. Everything beyond — the full logos strata, borrow checker, rewriting engine, optimization passes, standard library — is written in Logos and processed by the seed, until the system compiles itself. The seed's small size is what makes Logos formally tractable: auditable by hand, eventually verifiable, a fixed reference point. Once built it is compiled and immutable: its own logic is reflectable as Logic Graph — tooling can read it — but frozen, never rewritten at runtime, and it is the single trusted core every other part is processed by and traced back to. The discipline is restraint — knowing what belongs in the seed and what does not. The core the seed realizes is not bootstrapped sequentially: its identities — the synolon cell, `logos`, `Scope`, the primitives — are mutually dependent and come into existence together, one hand-built cyclic graph. Sketches of the core are therefore *specifications* of that graph, not programs (their textual order is not evaluation order), and checking the seed's graph against them is a planned verification step.

**The reflection boundary is callable-versus-data, and it bottoms out in `native`.** What reflects and what does not divides along *callables versus data*, not seed versus user. Every Logic Graph structure — logos, values, scopes, and a function's own source body — is reflectable, since reading it is exactly what the interpreter and the lowering do; what is opaque is the *callable*, a machine-code body (`@exec`) one invokes but cannot read into. This is forced, not chosen: interpretation consults each node's evaluator, so were every evaluator itself Logic Graph to interpret, the recursion would never bottom out — some functions must be machine code or nothing runs, and compilation exists precisely to produce them. A function is thus two things at once — a reflectable source body and, once compiled or native, an opaque callable — the same duality as interpreted code being its own Logic Graph while JITed code stays reflectable through the source it compiled from. A **`native`** function is the limiting case: callable-only, a machine-code body with no Logos source, the opaque leaf the tower rests on. The seed ships every primitive as `native`, which is what makes the bootstrap tractable — no spec-Logic-Graph is owed for the primitives, so the seed-against-spec verification above is deferred, not cancelled, returning per identity as that identity is ported. Self-hosting then replaces `native` primitives with Logos source one at a time, each gaining a reflectable body while keeping its callable, so the boundary never moves and the `native` set only shrinks toward an irreducible floor — `invoke`, allocation, raw memory, syscalls — that can never have a Logos source.

**The Logos IR is the codegen-oriented intermediate layer.** Between the Logic Graph and any backend sits a target-agnostic IR with explicit basic blocks, control flow, memory operations, and logos. Target-agnostic optimizations (constant folding, CSE, inlining, monomorphization) operate here. The Logos IR is itself representable in the Logic Graph — another stratum of the same structure. The Logos IR should be designed to fit many different backends, like Cranelift, LLVM, WASM, and potentially unique hardware. All of these parts are not the same set so there must be a way to describe what is possible and not for each backend. And the Logos IR should be extensible for the future. But the v1 Logos IR should at least be solid for most/all things Cranelift is solid at.

**The backend interface defines what a backend must provide.** Anything that consumes Logos IR and emits code — Cranelift, LLVM, a custom embedded compiler, WebAssembly, a GPU backend — satisfies one interface. Cranelift is the default (fast compilation, good code, ideal for meta-circular execution); LLVM is selectable for production and hot loops; backends mix within a program at function or module granularity via scope modifiers like `backend cranelift (...)` or `optimize_with llvm (...)`.

## The rewriting engine

Logos collapses three things other languages keep apart — compiler optimization, computer algebra, and user-defined transformation — into one operation: take a Logic Graph or Logos IR fragment, apply rewrite rules, extract the form minimizing a cost function.

The engine uses **equality saturation**: rather than rewriting destructively in priority order, it builds an e-graph of forms equivalent to the input under the rule set, then extracts the lowest-cost form. Extraction is optimal over the forms the e-graph actually holds; reaching a true fixpoint is not guaranteed — size-increasing rules can grow the e-graph without bound — so the engine runs under an explicit iteration/size/time budget and extracts the best form found within it. (Optimal extraction under a sharing-aware cost function is itself NP-hard, so the default extractor is heuristic, with an exact mode available.) The e-graph approach is production-validated — Cranelift's mid-end uses one — though production systems bound the search rather than fully saturating, as here.

- **Rule sets are first-class Logos values.** A rule is a pair of Logic Graph patterns (match, rewrite). The standard library ships canonical sets for arithmetic, algebra, trigonometry, and compiler optimizations; users compose, extend, or override them.
- **Cost functions are first-class Logos values.** A cost function maps Logic Graph nodes to costs. The standard library ships FLOP count, numerical stability, code size, register pressure, and target-specific metrics; users supply their own.
- **Multiple targets, one engine.** Minimum-cost (default `simplify`), canonical (equality testing), factored (root extraction), expanded (pattern matching), numerically-stable (Horner, log-sum-exp), and target-specific forms are each just a choice of cost function and/or rule subset.
- **Compile-time evaluation is first-class.** Because `eval` over Logic Graph fragments is built in, the engine can run at compile time on symbolic expressions and bake only the optimal form into the binary. Differentiating an expression, simplifying the derivative, and emitting native code is one `eval`-bracketed pass with no runtime symbolic machinery. The same applies to optimization: `eval (optimize_for(target, kernel))` performs target-specific rewriting on user code at compile time.

The same engine serves the compiler's `x + 0 → x` and the mathematician's `sin²(θ) + cos²(θ) → 1`. The compiler's optimization passes are Logos libraries over this engine, not hand-written compiler code; adding a pass is library work.

## The logos strata and verification

The strata compose, and users pay only for what they use.

- **Base logos stratum** — primitives, record logos, enums, arrays, function logos, references with lexical lifetimes, ownership and moves, capability annotations (`pure`, `total`, `async`, `thread_pinned`, GPU-subset, allocator-bound). The borrow checker operates here: for whole values and fixed places (fields, elements) it is decidable, fast, and points errors at nodes the user wrote; borrowing a *predicate-defined* set of indices is the advanced case, which discharges its disjointness obligation through the SMT layer and is therefore bounded by the solver, not constant-time. One rule covers every case: among references that are simultaneously live and overlap, many readers XOR one writer — read∩write and write∩write empty, read∩read free. The checker is therefore one generic function asking one question — *can these two extents overlap; does this access fall inside this extent?* — answered yes, no, or unknown; the tiers are not separate rules but answerers of rising strength for the same question (identity and field-offset comparison; constant then symbolic index arithmetic via SMT; predicate disjointness where decidable; kernel-checked proof beyond), and what varies between borrow-checking "cases" is never the rule but the difficulty of computing its inputs: which references coexist (lifetimes, control flow, threads) and which references exist at all (summaries at function boundaries, since references escape through data structures).
- **Refinement logos** — predicates on primitives (`i32 where >= 0`, `i32 where 0 <= self < array_len`), with an SMT solver discharging obligations at assignments and call sites. Rules out out-of-bounds, division-by-zero, and overflow at compile time wherever the solver can discharge the obligation; where it cannot, the obligation surfaces as an error to resolve — a strengthened precondition, an explicit check, or a handled case — never a silent pass. Code using no refinements is unaffected.
- **Pre- and post-conditions** — refinement predicates on function inputs and outputs; preconditions proven at call sites, postconditions in bodies. They compose: a call yields its postcondition as a new fact.
- **Termination annotations** — opt-in decreasing measures for functions that want to participate in proof contexts. Functions without them stay usable but cannot appear where totality is required.
- **Full dependent logos and proof terms** — logos depending on values, propositions as logos, proofs as programs, checked by a small trusted kernel. Arbitrary mathematical theorems about programs become statable and provable.

**Verification composes with rewriting.** Each rewrite rule can carry a proof that it preserves the relevant property (equality, refinement satisfaction, numerical equivalence within tolerance); the engine then guarantees simplification preserves it. A verified CAS — every simplification provably equality-preserving — emerges as a library.

A systems programmer gets the base logos stratum and borrow checker; someone wanting guarantees beyond memory safety opts into refinements and pre/post; scientific, cryptographic, or verified-systems work opts into dependent logos. The strata interoperate; parts of a program can be verified while the rest stays lower.

## The proof layer

**Proofs are a layer over the Logic Graph, not a property of the substrate.** The base node model classifies a value by following its `logos` pointer — a recursion that terminates in the self-referential `logos : logos`. That self-loop is sound as *operational* machinery (it is never read as a logic) but inconsistent as one (Girard's paradox), so the proof system does not build on it. Instead, propositions, proofs, and their classifying universes are ordinary Logic Graph values that **carry their logical universe as stored data, read by `universe_of` rather than `logos_of`**. The same nodes thus bear two classifications — `logos_of` for execution (self-looping, irrelevant to the logic) and `universe_of` for checking (stratified, stored) — so the proof system's soundness is decoupled from `logos : logos` entirely. This is the standard kernel treatment of universes (levels as checked data, as in Lean/Coq) recast in the node model.

**Universes are stratified; propositions are erased.** `universe_of` enforces a predicative tower — `Universe 0 : Universe 1 : …`, generated lazily, never `Universe : Universe` — which is what excludes the paradox. `Prop` is the distinguished universe of propositions: impredicative, proof-irrelevant, and erased before codegen, so verification costs nothing at runtime. A proposition is a value in `Prop`, a proof is a value inhabiting it, and the connectives (`False`, `And`, `Or`, implication, `Forall`, `Exists`, equality) are ordinary inductive and function logos — no proposition primitives. Curry–Howard holds at the *representation* level (a proof is a real, reflectable node carrying its logos) while logical classification is the separate `universe_of` relation, giving the isolation of a deep embedding without demoting proofs to inert data.

**The trusted obligations are small and explicit.** The kernel verifies each stored universe rather than trusting it (enforcing stratification), checks strict positivity on inductive definitions, and confines definitional equality to the terminating proof-layer fragment — the same tag that distinguishes a proof object from substrate junk gates what the checker may evaluate, so it can never be driven to reduce a non-terminating substrate term. Two walls follow: nothing eliminates from the `logos : logos` substrate into `Prop`, and conversion reduces only tagged, terminating objects.

*(June 2026 review — open decision. If `Prop` stays proof-irrelevant, one obligation is missing above: elimination from `Prop` into a computational universe must be confined to subsingleton propositions (Lean/Coq's large-elimination restriction), or proof irrelevance is unsound. The alternative under consideration is **proof-relevance** — proofs as ordinary, distinguishable, reflectable graph values, with erasure demoted from an automatic property of `Prop` to an ordinary optimization — which needs no such restriction and fits both the reflect-everything stance and the ecumenical layer below. Resolving this fixes the proof-irrelevance wording in this section.)*

**Theorems, axioms, and reach.** A theorem is a named proof — a definition whose logos is a proposition and whose hyle is a kernel-checked term. An axiom is a proposition declared with an empty hyle slot: asserted, not proved, enlarging the trusted base, with every dependent proof traced back to it. The sound layer can prove properties of *any* Logic Graph node — the self-referential substrate and the proof system itself included — by reasoning over representations and an operational model, treating substrate logos as syntax and never lifting substrate classification into logical truth (the CompCert/MetaCoq discipline). It verifies everything above it; by Gödel's second theorem it cannot prove its own consistency, which stays the irreducible part — trusted by hand-audit, not proved.

**The layer is deferrable.** None of this is needed for the substrate to run, so the proof system ships after Core, in the standard library. Its smallest checking core — the proof kernel — is an ordinary importable library, not part of the seed: it is trusted only by code that opts into it, so it never enlarges the seed's mandatory trusted base, which stays the single core everything is processed by. Everything stated and proved on top — the connectives, theorems, and tactics — is *checked* by that kernel, not trusted.

## The ecumenical proof system

*(Design direction reached June 2026; like the rest of the proof layer it ships after Core and is deferrable.)*

**Proofs do not live in one global theory; they live in a graph of theories.** A single consistent axiom set is one *branch*; the Logic Graph holds many, including mutually contradictory ones — `P` in one branch, `¬P` in another — each internally consistent, coexisting without poisoning each other. A contradiction arises only from combining incompatible assumptions in *one* proof, and the system never does: every proof carries the **set of axioms it depends on** (its branch), and two results combine only when the union of their axiom-sets is consistent. This is the **ecumenical proof system** — named after the proof-theoretic *ecumenical systems* in which classical and intuitionistic logic coexist in one framework — and structurally a *theory graph* (cf. MMT, Hets, the IMPS "little theories" method) that carries its own dependency provenance.

**The branch is the axiom-closure, not the whole dependency-closure.** Definitions are branch-neutral and shared; only *axioms* — propositions asserted with an empty hyle slot — commit a proof to a branch. So when the kernel checks a proof it emits the set of axioms that proof transitively rests on (as Lean's `#print axioms` does), and that set is the proof's world. Adding a branch *soundly* means forking on an **independent** statement — one the base can neither prove nor refute, like the parallel postulate over Euclid's other axioms, or CH over ZFC: you *assume* `P` or *assume* `¬P`, never prove both from one base (which would mean the base itself is inconsistent).

**Consistency is checked, never guaranteed.** Keeping `P` and `¬P` apart is trivial — one is the other's negation. But two axioms that look independent yet are jointly inconsistent derive `False` only through some chain, and finding that chain is undecidable (a system cannot prove its own consistency — Gödel's second theorem). So a combination is admitted as compatible **until a `False` is derived from it**, whereupon the cluster is marked poisoned and its results withdrawn — non-monotonic, in the manner of assumption-based truth maintenance and belief revision. "Consistent" means "no contradiction found within budget," never "proven impossible."

**A coherentist layer ranks plausibility, and it is read-only.** Backward search reduces a goal to the sub-statements that would establish it, recursively, and *where it bottoms out sets the status*: at axioms or proven theorems the goal is **proven** (certain); at a web of unproven-but-densely-interconnected statements it is **plausible** (a ranking, not a proof). The more consistent structure that depends on a candidate axiom, and the more it interconnects, the higher its plausibility — coherentism computed over the proof graph's own edges. These scores feed *what to try proving*; they **never** count as proofs — the same one-directional wall as the rest of the layer (kernel → ranking, never ranking → kernel). This is the system's humility: it self-corrects the instant a proof or a contradiction arrives, while keeping every still-consistent branch alive.

**It composes with the universe kernel rather than replacing it.** The two are orthogonal layers over the one graph: *universes* stop self-referential paradox **within** a theory (the stratified `universe_of` tower above); *branches* manage incompatible assumptions **across** theories. They meet at one productive point — large-cardinal axioms, which assert that taller universes exist, so there "which branch am I in?" and "how tall is the universe tower?" are the same choice, settled (as in real mathematics) by fruitfulness, i.e. by the coherentist layer.

## Memory and concurrency

- **Stack by default** — locals are scope-bound; their memory is reclaimed with the frame, an effect-free bulk operation. No GC. Reclamation and destruction are distinct: returning memory that needs no teardown stays implicit, while running teardown never is — teardown is always visible defer structure (next bullet).
- **Explicit heap, and no implicit destruction** — `alloc T v` returns an owning `ptr T`, and what frees it is never an invisible scope-end rule: **the constructor authors the teardown and inserts it as reflectable `defer` structure**. Only the construction site knows how a value was built — which allocator, whether ownership was taken, what invariants hold — and a `ptr T`'s logos cannot carry the whole story, since ownership does not always follow the name holder (sometimes the pointee must survive the name). So `alloc` inserts `defer free a` — the *specific allocator* captured in the inserted call, which is what lets many allocators coexist safely (see *User-replaceable allocators*) — into the scope where the result's ownership lands; and `alloc` is only the first instance of the general rule: *any* constructor may insert its paired teardown (a file-opening constructor inserts its `defer close`), and a record logos's auto-derived constructor composes the teardowns its field definitions declare, because deconstruction has too many shapes for any single mechanism to know them all — each constructor knows its own. `defer` runs LIFO at scope exit, so teardown order reverses construction order. `drop x` applies to an identity and runs its destructor. `drop` and `own` are `take`s (the access kind below: consume, emptying the source to `undefined`): a pending deferred teardown over an emptied place is a *sanctioned no-op* — the phase bits are the drop flag — which is what cancels the inserted defer on an early `drop`, on ownership transfer out of the scope (`own` transfers ownership and removes the source identity), and on a conditionally-moved value, all with no graph edit; the compiled tier elides the flag check wherever the checker proves no conditional move, as Rust elides drop flags. Because teardown is ordinary graph structure rather than hidden drop glue, "every alloc reaches a paired free on every path" is a provable proposition over the graph, discharged by the same proof layer as everything else — RAII as checkable structure, not a trusted builtin. `&T` / `&mut T` are statically checked (many shared OR one exclusive), with lexical lifetimes keeping the model predictable. Recorded as rejected, to stay rejected: implicit scope-end destruction in any form, and `:=` inserting `defer drop` on every declaration — declaration stays neutral; teardown authorship belongs to constructors alone. *(Ruled in discussion, July 2026. Still to spec: the exact attachment rule for temporaries — a constructor result passed directly as an argument — and the composition order of field-declared teardowns.)*
- **Place-granular borrows** — borrows are tracked per *place* (a field, an element, a predicate-defined set of indices), not per whole object, so the borrow rule (many shared, one exclusive) applies only where two places provably overlap — a disjointness obligation the checker discharges through the refinement/SMT layer, being itself a Logos program free to query the solver. Exclusive and shared borrows of disjoint regions of one object therefore coexist — odd indices writable while even indices stay shared-readable, say — capturing safely the split-borrow patterns conventional checkers reach only through `unsafe`. These borrows are *local*: resolved at the borrow site, within the scope that can see the place, and they do not outlive it carrying their extent — the escaping case, where the extent rides on the reference identity itself, is *Reference-granular permissions* below.
- **Reference-granular permissions (post-v1; v1 must not block them).** The fine-grained extent of a borrow is carried by the *reference identity itself* — a borrow is already its own node — as a total, pure predicate over places. The obligations it generates (disjointness between coexisting references, containment of each access) discharge statically through the tiered ladder above, and predicates and proofs erase before codegen like everything in `Prop`, so a fully discharged reference compiles to a bare address; a runtime check is only ever an explicit opt-in fallback, never silently inserted. What v1 owes this future: references exist as graph identities at check time and erase to raw addresses in codegen; the extent descriptor is deviation storage (the default whole-place reference stores nothing); the checker is factored around the one overlap/containment query so later tiers plug into its *unknown* bucket; places are graph values a predicate can mention; the optimizer keys aliasing and immutability on a reference's extent, never implicitly on whole objects; and extent predicates are admitted under the same obligations as recognizers — total, pure, terminating.
- **Read and write are one mechanism across the system.** Visibility (`pub`), mutability and borrowing (`&` / `&mut`), and graph access (reflection and `graph_mut`) are not separate features but one primitive — a read or write *capability over a place, for a lifetime, granted to a scope*. Visibility is the `'static`-lifetime read of a name; a borrow is the region-lifetime read or write of a value or sub-place; reflection and graph mutation are the read and write of a graph node. The borrow rule — many shared XOR one exclusive — is the reader–writer rule, and it governs all of them uniformly, including across threads. The scope tree supplies the default grants (a declaration is reachable in its own scope and descendants), so the common case needs no annotation and `pub`/`mut` remain sugar over the primitive. One model, two enforcement timings: static checking for *accessing* fixed structure, incremental re-validation for *mutating* it — because writing the graph changes the structure the checker reads. The checker holds a privileged read over the whole graph — it must see every node to identify and check what is built on it — so user-facing visibility restricts *peer* access, never the checker's reflection (a private field is hidden from other code, not from the checker). Beyond the coarse per-branch grants the scope tree gives, access may also be set by *functions* — total, pure predicates over places or scopes — for the fine-grained cases (these indices writable, that field readable only by its logos's methods). These access-setting predicates are **`gate`s**, ordinary identities like any other — `pub` and `mut` are themselves gates. A gate is a **fail-closed predicate** returning `true | false | ?` (the universal unknown): access is permitted only on `true`; `false` and `?` both deny, with `?` retained as a standing obligation — the synthesis goal a later prover discharges, moving an access from `?` to `true` by proving it safe (composition is Kleene three-valued, so an unresolved `?` stays visible, and only the enforcement boundary collapses `?`→deny). The check is **lexical and static** — decided from the viewer's scope chain at elaboration time and erased before codegen — so there is **no ambient authority**: being *called by* a holder grants nothing, an indirect call cannot launder access, and the only way to give a callee access is to hand it the reference explicitly. A gate's requirement propagates *backward* through the call graph as a static obligation, discharged at the scope that legitimately grants it, and it stays analyzable starting from any code section. The access kinds themselves are an **open set**, not a substrate enum: `read` and `write` (observe, mutate in place) are joined by `call` (invoke an authority without inspecting it) and `take` (consume/move, emptying the source to `undefined`), with more — borrow, alias, drop, observe-for-proof — admitted the same way. A gate computes the kind in force by analyzing the live evaluation stack; what keeps that sound is that each operator declares an **effect signature** classifying its operand slots, and the trusted evaluator stamps the access mode from that signature — evaluator code cannot forge a read into a write. Gates run on the *same* stack they analyze (a separate meta-stack was rejected as a regress), bounded by one rule: everything above the first gate-evaluation marker on the stack runs ungated, so gate evaluation terminates. *(Mechanism from design discussion, June 2026; not yet validated.)* The seed builds the chokepoint, the lexical tier, and the classified `?` seam only — **no prover**; the stronger answerers (the borrow checker, the proof layer) bolt onto `?` exactly as the tiered checker above describes.
- **Reflectability is a droppable capability, so the resident graph is a dial.** An interpreted region *is* its LG and a JITed region stays reflectable *through* its LG (see *The architecture*), so keeping a region reflectable means keeping that LG resident — which is the only thing the source-to-graph size expansion actually charges for. But reflection is a read over graph nodes, governed by the one reader–writer rule like every other access, so it is *revocable*: drop the last read capability into a compiled region and its LG has no reader, is reclaimed by the same reachability that frees a scope's runtime graph (*Two graphs, one source of truth*), and the region collapses to its bare callable — a `native`-like leaf, an `@exec` body with no readable source. This adds no collector; sealing a region is the reader–writer rule applied to the graph's own memory. Resident cost is therefore a spectrum the program sets per section: fully reflectable — the whole image, seed and stdlib and compiler and user code all resident as LG with logos and proofs attached (the Smalltalk/Lisp-image reality, heavier here for carrying the proofs); seal the toolchain once booted — drop its LG, keep the callables — and the resident set collapses to what is still reflected on; seal everything and the graph is gone, an ordinary native binary at no graph overhead. The expansion factor is thus not a fixed tax but the price of retained reflection, paid only where reflection is kept and reaching zero as regions seal. The seal is a contract, not a free lunch: it keeps what other code has already read out — logos, compiled form, a proof's checked certificate — and forfeits the right to re-derive them (recompile, re-prove, structurally edit, or introspect that section), the same shape as stripping symbols from a release build; and only a section's *private* nodes are reclaimed, since interned identities and any subgraph another live reader still points into stay pinned, so the reclaimed fraction is how much of the section was private rather than shared. *(Design direction reached in discussion, July 2026; not yet validated. "Seal" is a placeholder name for dropping a region's reflectability.)*
- **User-replaceable allocators** — `alloc.arena`, `alloc.bump`, `alloc.pool`, `alloc.system`, and custom allocators share one interface. Arenas enable cyclic structures via indices rather than borrows.
- **Two concurrency shapes** — `parallel for` for data-parallel work over disjoint indices (the borrow checker recognizes the pattern); async for I/O-bound concurrency via stackless tasks on user-defined executor pools. A function that can pause is marked `async` and pauses at an explicit `.await`, so suspension is always visible in the source, and `.await` is allowed only inside an `async` function; the `async pool (...)` modifier selects which executor pool runs a region. Tasks are anchored once started (no Pin machinery); borrows cannot cross awaits except within the same task's anchored state record; capabilities flow through the call graph to enforce this statically. What a task may park on is an open set, so the event-listening layer is user-definable the way allocators are: one small interface — register interest, promise a wakeup — with implementations as ordinary replaceable identities, the waker being nothing but "make that task runnable again," and the reactor living with the pool definition it serves (an io_uring pool, an epoll pool, a GUI pool whose reactor is the platform message loop; GPU fences, embedded interrupts, and chain confirmations enter the same way). The stdlib ships the standard pools, so naming one is the baseline and writing one is the escape hatch. *(Ruled in discussion, July 2026.)*
- **Scheduling, preemption, and cancellation.** A task yields cooperatively at each `.await`; beyond that, the runtime can step in at any *boundary* where the task's live state is materialized as graph data — between interpreter steps, or at the edge of a compiled burst. This is what keeps one task from freezing its pool: instead of one opaque compiled function holding state in registers across the whole task, a hot region may compile to short bursts that read their inputs from the task's live state, run at full speed, and write results back at each boundary, so that state is ordinary graph data at every boundary. Preemption and cancellation are then uniform and cheap — a paused or preempted task *is* its materialized state, with no stack maps to emit and nothing to reconstruct from registers; cancelling is dropping that arena-scoped state like any other runtime graph, running the pending `defer`s of its live scopes (teardown is always defer structure — see *Explicit heap, and no implicit destruction*). Only the transient within-a-burst work is ever held in registers, and it is never seen from outside because control is taken only at boundaries. Boundary granularity is a per-region dial: short bursts give responsive preemption and cancellation, at the cost of writing live state out each boundary rather than keeping it in registers and of optimizing only within a burst; one large burst — a whole hot loop — gives top throughput but few stopping points, reserved for compute kernels, the longest burst bounding the worst-case time-to-preempt. It rides on the same interpret/compile boundary the representation section describes (a compiled region falling back to interpretation): where an artifact can deoptimize is where the scheduler regains control. Preempting is invisible and the task resumes seamlessly; cancelling stops it at its last boundary for good — its values are dropped but its remaining steps never run, so keeping state consistent across a possible cancellation is the program's responsibility. Two rulings bound the dial. Splitting a region into bursts is never hidden logic: boundary points are placed by the user or by an ordinary identity — a backend or pool policy — and what gets inserted is reflectable structure, the same law that governs constructor-inserted teardown; the compiler never adds yield points behind the graph's back. And a pool's latency property — no burst here over a millisecond — is not a static WCET proof, which exists only for heavily restricted code: it is empirical, measured history combined with the region's structure, and enforced by deoptimization — a burst that blows its budget is an observable event that demotes its region to finer bursts or to interpretation, so the bound is self-correcting rather than proven. Both belong to the adaptive layer, after v1. *(Ruled in discussion, July 2026.)* *(Scheduling model reached in design discussion, not yet validated; the bootstrap can ship cooperative `.await` yielding and add boundary preemption as the compiled tier matures.)*
- **Case study: making an RCU library's comment-enforced rules machine-checked.** [TSM](https://github.com/ThobiasKnudsen/LogosMath/blob/gd_to_tsm_and_gtsm/include/tsm.h) is a lock-free, RCU-protected, type-generic hash table in C (userspace-RCU LFHT under a node model where every node's type node carries its `fn_free_callback` and `fn_is_valid` — a C precursor of the synolon), and every safety rule it imposes on callers, it states only in doc comments. Each class of those rules maps onto a Logos mechanism that enforces it. *Conditional ownership transfer* — create/insert/update "take ownership … so you should not free … if successful" — is exactly `take`: on success the callee empties the caller's place to `undefined` and the caller's constructor-inserted `defer free` no-ops on it; on failure ownership stays and the defer fires — the leak and the double-free both become unreachable instead of comment-warned. *"Free only via `call_rcu`"* — the node constructor inserts the RCU-deferred free as the only teardown path, RCU-deferred freeing being just another allocator behind the one allocator interface. *Forgotten `rcu_read_unlock`* — the read-guard's constructor inserts `defer unlock`. *Use-after-grace-period* — "a pointer obtained inside the read section must not be used after it" is not a runtime-insertion matter but a lexical lifetime: the reference's lifetime is bounded by the guard's scope, the borrow checker's ordinary static territory — the library's hardest rule happens to be lexically shaped, so the lexical-lifetimes model fits it exactly. *"Never `synchronize_rcu`/`rcu_barrier` inside a read section"* (and defer-free must run outside one) — effect rules, enforced by capability tracking: those operations demand a capability the guard's scope revokes. *A type node must outlive the nodes it types* — an ordinary lifetime relation between identities. The interpreted tier catches the dynamic classes at runtime as checked errors, never UB; the static tiers (borrow checker, capabilities) close aliasing and effect ordering; together, 100% of the header's stated obligations are machine-enforced where the C original enforces none of them.

## Mutability and construction

*(Mechanics settled in design iteration, June 2026; declaration syntax and the `mut` model revised later that month — `mut` recast as a logos modifier (mut on the logos denotes a mutable value), not a positional slot qualifier — applied recursively up the classifier tower, with hyle- and logos-mutability as independent permissions on the two slots. The phase-bit storage is proposed, not yet validated, and whether everyday hyle-mutation rebinds to a fresh synolon or writes in place is still open.)*

**Declarations are immutable by default; the grammar mirrors the synolon's two slots.** A synolon *is* a logos slot and a hyle slot, and three operators write them: `:` is **declaration** — it introduces a name and sets its logos slot (`key : logos`), leaving the hyle `undefined`; `:=` is **declaration with a value** — it introduces a name and sets its hyle slot, inferring the logos from the value (`key := value`); and `=` is **reassignment** — it writes the hyle slot of an *already-declared* name (`key = value`). The colon writes the logos slot, the other two write the hyle slot; they do not compose — `key : logos = value` is rejected. A name declared with a logos takes its value by later reassignment (`key : logos`, then `key = value`), and a value with its logos is written by juxtaposition (`key := i32 32`). The two declaration operators differ in what they accept: `:` requires its operand to *be* a logos (that is what a logos slot holds), while `:=` accepts **any** synolon as the value — a logos included, since a logos is a value like any other. `x := i32` is therefore legal and binds the name to that logos itself, making `x` another spelling of `i32`, usable wherever `i32` is; and because a logos's own hyle slot starts `undefined`, `a := i32 ?` and `a : i32` declare the same synolon. A bare `key :` declares a name whose two slots are both `undefined`, and an anonymous value with its logos is written by juxtaposition — `i32 32`, `f64 2.0`, the logos preceding the value — never `logos = value`. The relation is exact: a name may carry a logos and a hyle but need not, a logos may carry a name and a hyle but need not, and **a hyle always carries a logos** (the logos defines how it is read). Names live in the trie; logos and hyle live in the synolon.

`=` is **reassignment, not declaration**: `key = value` overwrites the hyle of an *already-declared* `key`, keeping its logos, and is an error if `key` was never declared — so a mistyped `cont = 6` on an undeclared `cont` is caught rather than silently creating a binding. Declaration is therefore always `:` or `:=` (`key :`, `key : logos`, `key := value`); `=` only ever rebinds what already exists, and logos inference rides `:=` (the logos is read from the value). The three operators sit at the very top of the precedence order — the loosest binders — and associate right to left, which is what lets the language define its own operators: `:= := …` is a well-formed declaration whose left operand is the `:=` spelling itself. Parentheses are dual in that order: the grouping paren binds loosest of all, the call paren tightest. *(Settled in discussion, July 2026.)*

**`mut` is a logos modifier — `mut T` is the logos of a mutable T**, so mutability is something the *logos* states about the value, not a separate value-position qualifier. The logos is the sole authority on what the hyle permits, so the capability lives there: a value whose logos is `mut T` is the everyday mutable variable, one whose logos is `T` is immutable, and there is no value-position `mut` form — `mut` never sits before the value (`i32 mut 32` is rejected: the hyle is raw bytes, with nowhere to carry the capability). This is the very same `mut` as in `&mut T` (mutable access to a T), one rule rather than two, which also retires the old inconsistency of writing `mut` on a reference logos but forbidding it on a declared logos. A record logos's per-field mutability rides on each field's logos the same way (`f : mut F` mutable, `f : F` not), so the logos alone describes which parts are writable. Changing the logos slot itself — reclassifying the synolon rather than writing its hyle — is not a hyle-level mutation: it needs the logos's *own* classifier to carry `mut` (the identical rule one level up), or a fresh `synolon` built with the new logos, which is the immutable-by-default path. This is one recursive rule: **`mut` left-prefixes a logos at any level of the classifier tower (`value : i32 : logos : …`) and makes the level just below it mutable** — `mut i32` marks the *hyle* mutable, `mut logos` (mut on the logos's own classifier) marks the *logos slot* mutable. The two are independent permissions on the two slots, so a value-construction expression spells out four states: `i32 32` (frozen logos, frozen hyle — the plain constant, by far the most common), `mut i32 32` (frozen logos, mutable hyle — the ordinary variable), `mut logos i32 32` (mutable logos, frozen hyle — *reinterpret-only*: the fixed bytes may be re-read through another logos but never overwritten), and `mut logos mut i32 32` (both — full structural mutability). Making the logos mutable does not imply a mutable hyle; reinterpret-only is its own quadrant. The tower tops out at the `logos : logos` self-loop, so `mut` sits at only these two levels in practice — anything higher would mutate the universe. Prefixing the *name* with `mut` (`mut key`) still makes the **name/identification** mutable — the trie entry, not the synolon, since the synolon's content is already reachable through its two slots — and `mut (…code…)` still makes a whole region **structurally** mutable. These stratify by how much structure can change, which is exactly the compile/freeze rule: hyle-mutation is ordinary data change and stays **compilable**; a logos change rewrites how the hyle is read, so it is **checked and deoptimizes** on write; a structural `mut (…)` region has a live structural writer and therefore **stays interpreted** until it freezes.

**Both slots of a synolon follow one lifecycle: `undefined` → defined → frozen.** `undefined` is the hole — no value yet, writable until defined, and reading it is a checked error, never undefined behavior. The exclusivity rule makes that check local: while a slot is still mutable, only the constructing scope can reach it, so no other scope can ever observe an undefined slot — the flip *is* publication. The two slots obey **one rule at successive levels**: each is written by its definition and then frozen by default, and stays writable past definition only when its classifier carries `mut` — the hyle while its logos is `mut T`, the logos slot itself only while the logos's *own* classifier is `mut` (the same modifier one level up). The giant common case carries no `mut`: defined once, frozen, the logos reading true forever. Reclassifying a live synolon is therefore not an everyday declaration — it needs that one-level-up `mut` or a fresh synolon — and it is not free: a logos write must discharge **validity** (the existing bytes are a valid value of the new logos) and **layout** (the new logos reads the same shape, or storage is reallocated), obligations the refinement/proof layer discharges where it can and that fall to `unsafe` where it cannot, exactly like a checked `transmute`. Because reallocation is itself a hyle write, a *layout-changing* reclassification additionally requires the hyle to be mutable; the mutable-logos/frozen-hyle state therefore admits only layout- and validity-preserving reclassifications — the bytes are re-read in place, never moved or rewritten. Structural graph edits that rewrite a synolon's logos are the other sanctioned logos writes, triggering re-validation as all `graph_mut` writes do.

**The flip is `immut x`, and it is one-way.** Flipping is a write to the synolon's phase, so the ordinary reader–writer rule blocks it while any reference is outstanding — making it owner-only with no live borrows, the same shape as a move — and no separate authority mechanism is needed (a `metamut` grant was considered and rejected; delegating flip authority is just ownership transfer). The reverse flip is rejected outright: facts derived from reading a frozen synolon outlive any reference to it — a checker concludes and drops its reference — so un-freezing is unsound without universal dependency tracking. Frozen is final, which is also why a flip never invalidates anyone: no reader can have assumed less mutability than existed.

**Phase is stored as tag bits, not wrapper nodes** *(proposed)*. A wrapper around the synolon breaks logos transparency — reading the synolon's logos would yield the wrapper — and a wrapper spliced into the hyle chain falsifies the prime invariant that the logos defines how the hyle is read. So the construction phase lives in neither: synolons are sixteen-byte aligned, the low bits of the logos pointer are therefore free (four of them), and they record each slot's construction state — undefined, defined, or frozen — for both the logos and the hyle slot; name-mutability and structural `mut (…)` are not per-synolon phase, living instead in the trie entry and the region marker respectively. The flip clears a bit in the synolon's own cell — no wrapper ever exists, no address anyone holds ever changes — and the model stays uniform while the storage stays free: every synolon *has* a phase, none *stores* one as structure. Interpretation masks the bits; compiled regions are frozen by construction, so masking is elided exactly where speed matters. Whether the phase should also be *visible* as structure is a presentation question, not a storage one: reflection can synthesize a wrapper view of a synolon whose mut bit is set, so tooling that wants mutability in-your-face gets it without every synolon under construction paying sixteen bytes for the privilege. Per-field mutability of instance fields is carried by neither bits nor wrappers — fields in a hyle area are raw bytes at offsets, not synolons, under any scheme — but by each field's declared logos in the logos's scope (`mut F` vs `F`), per the modifier above.

**Two muts, and the storage partition.** The `mut` in a *declaration* is a per-field fact across all instances: it is what the derived layout reads, it is fixed when the logos is defined, and an instance's later flip never changes a shared layout. A field that is ever mutable — even only during `init` — is per-instance; a binding that is never mutable is identical across every instance and is stored once with the logos. A record logos is thereby simultaneously C++'s struct (its mut declarations, from which the per-instance layout is *derived* rather than authored) and C++'s namespace (its immutable members, stored once) — and a logos whose members are *all* shared is a pure namespace, which is what a language is: the merge of `type`, `struct`, and `language` into the one identity `logos` *(ruled July 2026)* is this partition read to its end. Placement is decidable at the declaration site, writes being the only thing that can differentiate instances. A logos's layout-relevant slots must be defined and frozen before its first instantiation; the logos locks there.

**Resolution is one rule.** A logos keeps the defining scope it was constructed from, and `a.x` asks `a`'s logos for `x`; the declaration found decides everything — declared mut: a *place*, the byte offset inside `a`'s hyle area; immutable: the stored synolon itself. Visibility is one extra test at the same lookup. A call frame is an instance of its function — parameter declarations resolve to a frame exactly as field declarations resolve to an instance — and lookup continues up the logos chain, terminating at the `logos : logos` self-loop.

**Substrate vocabulary.** The cell is the **`synolon`**, its two slots the **`logos`** and the **`hyle`** — form, matter, and the compound, one doctrine in three words *(ruled July 2026; renamed from the earlier dyad/type/value)*. The classifier is **one identity, `logos`**: the former `type`, `struct`, and `language` were three defaults of one thing — a record logos derives its layout where a bare logos leaves parsing and semantics to its author, and a logos whose members are all shared is a namespace, at the limit the language itself — so all three merged; `type` and `struct` may return later as stdlib alias spellings through the ruled `x := i32` mechanism, never as separate core identities. The identity `logos`, the `logos` binary, and the `.logos` extension share one spelling deliberately: every classification chain terminates at the `logos : logos` self-loop, so the name of the language is the name of the ground everything in it stands on. `any` was removed and split: `undefined` is the hole, carrying a definedness obligation; `?` is the **unknown** — a deliberate, classified hole the author writes in place of code not yet designed, inert until filled and rendering any build that reaches it incomplete by construction; it differs from `undefined` (which says *nothing was written here yet* — a checked error to read) by saying *the value is not yet known*, a standing synthesis goal a maturing Logos discharges by the same backward proof search and coherentist plausibility ranking the proof layer already provides, graduating a `?` from left-alone to suggested to filled-and-verified. `@void` is a logos-erased address whose interpretation comes from elsewhere — the honest logos of the synolon's hyle field, since interpretation always comes from the logos pointer. `native` is a function that is callable-only — a machine-code body (`@exec`) with no Logos source, opaque to reflection (invoked, never read into), and distinct from both `undefined` and `?` because it runs and reaching it is no error; the seed's primitives are all `native`, ported to reflectable Logos source one identity at a time (see *The architecture*). Recorded as rejected, to stay rejected: `metamut`, the reverse flip, `any`, separate `type`/`struct`/`language` core identities (merged into `logos`, above), the plural "logoi" in any Logos writing (logos is invariant; a synolon pluralizes as synolons), a per-record field-name hashtable (field names live in the shared name index, scope-filtered — one resolution mechanism), inheritance, imitating `while` with a stepped `for` (a per-iteration condition deserves its own spelling), and the composed declaration `key : logos = value` (the declaration forms are `key :`, `key : logos`, and `key := value` alone).

## Native interop and the Rust bridge

**Logos reaches existing native code as black-box C-ABI calls, not by hosting other languages.** The runtime is an ordinary Rust program built by rustc (LLVM); Cranelift's role is to JIT-compile Logos code at run time. A foreign function is called in-process, by address, across the platform's native calling convention — the same convention `extern "C"` selects, and the one Cranelift-based JITs like wasmtime reach (through trampolines) to call host functions — so even JIT-compiled Logos code calls native code with no linker step, independent of which backend compiled either side. The boundary is the conventional one — `extern "C"` signatures, `#[repr(C)]` data, panics caught before they cross, an `unsafe` contract the borrow checker cannot see past — and richer Rust types (`Vec`, `String`, generics) are reached through a thin per-function shim a binding macro can generate.

**The toolchain is a build dependency, not a runtime one.** Logos ships as a standalone interpreter; that it is written in Rust is how it was built, not what it needs to run. Native code is compiled in as core builtins, loaded at runtime as separate `cdylib` artifacts (interpreter and Rust toolchain ship apart), optionally produced on demand by invoking `rustc` and then loaded, or isolated behind IPC where a crash must not propagate. Hosting a language's full semantics as a Logic Graph dialect remains a separate, heavier option; the bridge is the cheap path to reusing existing ecosystems.

## Error handling

Errors are values: tagged unions `(T | Error)`. Fallible functions declare their error logos; callers handle results explicitly via `match`, with `success` / `fails` combinators for boolean outcomes. No exceptions; no implicit propagation in the core, though a propagation sigil may come as sugar.

## Meta-reflection and the unified system

This commitment ties everything together.

- **The Logic Graph contains its own parsing rules.** Every token, operator, precedence, and associativity is a queryable node; the parser is non-context-free because it consults the trie, which is itself part of the graph. Tooling becomes language-aware by introspecting the graph rather than reimplementing the syntax.
- **The Logic Graph contains the compiler.** Above the bootstrap seed, the borrow checker, logos checker, rewriting engine, optimization passes, and the lowerings (Logic Graph → Logos IR → backend) are all Logos programs and all graph subtrees; the seed itself is the frozen exception — reflectable but not rewritten. Adding an optimization is library work; targeting a platform is implementing the backend interface plus contributing rules.
- **The Logic Graph contains the standard library.** Standard-library code is not special; it is code that ships with the system, readable and (carefully) overridable.
- **The Logic Graph contains documentation, source positions, comments, and metadata.** Docs and comments are the same thing: `#`-built prose nodes interleaved with code, void-valued and invisible to value flow (see *Text literals are plain values*); cross-references are graph edges; positions are node properties (held in the derived source-map).
- **The Logic Graph contains proofs and specifications.** A refinement is a subtree of a signature; a correctness proof is a subtree of a definition. Refactoring code and refactoring its proof use the same tools.
- **The Logic Graph eventually contains the IDE.** Editor behavior, highlighting, refactoring, debugging UI, and configuration are Logos programs over Logic Graphs — hot-reloadable, customizable in Logos, with structural editing as library functions. The Smalltalk vision applied to a modern systems language.
- **The Logic Graph eventually contains the running application, and the browser.** Because the IDE is Logos over the graph, and Logos has I/O and compiles to portable backends, the same reflective structure grows without a rewrite into a shareable, networked, rendering platform: a browser reached from the tooling end rather than built as one. See *The IDE as platform, and the reflective web*.
- **Stratification handles foundational paradoxes invisibly.** When the logos strata reason about themselves or proofs quantify over the proof system, a universe hierarchy (following Lean 4) keeps things consistent. Ordinary code never encounters it; only explicit meta-mathematics does.

## Representation, elaboration, and lifetimes

This section records the mechanics of how the Logic Graph is represented, how source becomes runnable, and how runtime-generated structure relates to source.

**A synolon is a logos and a hyle; its definition supplies everything else.** Every synolon is exactly two pointers — a `logos` and a `hyle`, sixteen bytes — and a synolon's *identity is its address*, so synolons are uniformly addressable by handle whatever allocator holds them (no arena is required; the address is the id). The `logos` pointer defines how the `hyle` is read: the logos's *definition* — itself a Logic Graph — specifies the synolon's layout, how it consumes surrounding tokens during parsing, the conditions under which it lowers to Logos IR, and the IR it produces for each well-checked case (a `+` over mismatched or sizeless logos simply does not lower until that is resolved). Operands are *not* stored inline; they are reached through the `hyle` pointer, so a binary `+` is a single 16-byte cell referencing its two operands rather than three inline pointers. What `hyle` points at is an operand *record* whose layout the synolon's definition supplies — the same rule that a logos defines how its hyle is read, applied to operators: a binary operator's hyle is a two-field record (LHS, RHS) that a third field extends once its operand logos resolves — the conserved `[LHS, RHS, logos]`, so run, compile, and reflection dispatch on the stored concrete op rather than re-derive it — a function call's is its argument record (one field per argument, a one-field anonymous record for a single argument), and a variadic call's is that record sized per call site. A growable list is never inline either: a sequence's expression array rides behind one indirection, an `array` value the scope synolon's hyle references. The operator is thus a higher-level identity describing *how to read its operands*; which concrete machine operation runs — which `+` — is resolved from the operator together with the operand logos, not from the `+` synolon alone. A synolon is therefore directly addressable once its handle is held, but the graph is *not* addressable by position or name without context: source code is meaningless without its surrounding context, so lookup from text proceeds by trie-directed traversal (the same trie the non-context-free parser consults), with auxiliary indices added where repeated reflection warrants them. Every synolon *is* an identity; most identities carry no string name and are addressed structurally, by pattern-matching over the trie and graph, rather than by name. This is MLIR's idea — a node whose *definition* describes its parsing, operands, and lowering — recast onto a uniform two-pointer cell and fused with the reflective parser. MLIR itself is a candidate *backend* behind the backend interface, never the IR: the Logos IR must be a Logic Graph stratum so the same rewriting engine, logos strata, and proofs apply to it, which an opaque external IR could not provide.

**Numeric literals are uncommitted until context classifies them.** A numeric literal lexes as `rational_number` — not a fixed machine logos but a comptime-number that can mold into any `int`, `uint`, or `float`. Because the concrete `+` is chosen from the operand logos (above), a literal beside a classified value takes that value's logos — `a + 1` with `a : i32` is i32 addition, `1` molded to i32 — which is what lets one operator and one literal form serve every numeric logos without per-literal suffixes (`1i32`, `1.0f`). When *both* operands are still `rational_number`, there is no machine logos to mold to, so the operation is carried exactly over the number's reduced fraction (`[num, den]` — a rational is a fraction pair, never text) and the result *stays* `rational_number`, committing to a concrete logos only when it finally lands in a classified slot. The resolution rules follow from this: two operands of the same concrete logos keep it; a concrete logos beside a literal molds the literal to it; two literals stay `rational_number`; two *different* concrete logos do **not** silently lower — there is no implicit coercion. Crossing logos is therefore explicit, and reuses the token-consumption mechanism above rather than a separate `cast` identity: a numeric logos applied to a value (`i32(a)`) *is* the conversion, per-constructor rather than a uniform cast syntax. A comparison resolves its operands the same way but its result is `bool`, one of the primitive logos. The logical operators short-circuit; the seed's `not` takes a parenthesized operand (a parsing simplification, not a keep). Integer division and remainder are **total, saturating**: `x/0` and `x%0` yield the logos's MAX, and signed `MIN/-1` saturates (`MAX` for `/`, `0` for `%`) — a deliberately loud value, easier to discover than a silent `0`, and the integer analogue of float division's infinity; float `/` is IEEE, and float `%` does not exist (rejected at parse). That is the base tier's defined behavior where nothing proves the divisor nonzero — division by zero cannot be ruled out statically in general — while the refinement tier turns it into a compile-time obligation wherever guarantees are opted into; the two layers compose rather than conflict. The seed folds both-uncommitted arithmetic exactly over `i64` fractions (an out-of-range result is a clean error, not a wrap); true arbitrary precision is deferred, not blocked.

**Text literals are plain values; `#` is the one comment constructor, over the same substance.** A string literal is written `«…»` and is always an ordinary value — never position-dependently a comment, so no value rule ever has to guess what a bare string means (a docstring convention was considered and rejected: at an untyped tail, `x := ( 40 + 2, «a» )` cannot decide between doc and value). Prose is instead always *marked*: `#` is an ordinary identity whose constructor consumes either a following `«…»` string or the raw text to the end of the line (the line form is sugar for the string form), and builds a **comment node** — the same text substance either way, one token, no line/block/doc-comment zoo. A comment node is real, reflectable graph structure interleaved with the code it annotates — documentation is therefore extracted by graph query, not by scraping special syntax — but it is void-valued and invisible to value flow: evaluating one yields nothing, and the value rules (a scope's trailing expression, tail commitment) read through it, so a trailing `# done` never becomes a scope's value and prose never returns anything to the enclosing scope. The seed already builds the reflectable comment node over the `string` substance, in both forms: `# raw text` to end of line, and `# «…»` bounded by the string alone — spanning newlines and ending at its `»`, with anything after it live code; `#` becomes a graph-resident constructor identity at self-hosting. *(Settled in discussion, July 2026.)*

**Pointer logos are prefix `@T`; dereference is postfix `x@`; `&x` is address-of.** *(Settled in discussion, July 2026; supersedes the earlier postfix spelling `dyad@`, written when the synolon was still called the dyad.)* The pointer is the first thing the user interacts with, so it reads first: `@i32` is a pointer to an i32, composing as `@@i32` and applying to any logos (`@point`, `@synolon`). Dereference puts the pointer on the left — `p@`, chaining as `p@.x` and `p@@` — reading in evaluation order where a prefix form would need parentheses (`(@p).x`). The split is also what keeps the grammar unambiguous: a dereference can never *start* an expression, so an `@` after a completed value is always a dereference and an `@` opening a logos position is always the pointer-logos prefix. `&x` takes the address of a storage-backed place (`&p.x`, a field's). The `@`-family is the *raw address* stratum — unchecked, which is all the seed's v1 provides — and the borrow checker's checked references (`&T`, `&mut T`) later layer on top of it: the same addresses, plus the proof they are used safely.

**The synolon is self-describing down to one fixed point.** The synolon's `logos` field is itself declared as a pointer to a *synolon* — not to a classifier primitive: that something functions as a logos is a checked invariant, not a substrate field constraint, because the substrate cannot state it. Interpretation is the recursion — to read a synolon, consult its logos; to read that, consult *its* logos — grounding out at the one synolon whose logos is itself, whose layout is the seed's only a-priori knowledge. With the hyle field a logos-erased address (`@void`) and the logos field a synolon address (`@synolon`), the substrate is described purely in terms of itself plus raw addresses.

**Source becomes runnable by interpretation; compilation is opt-in.** Source text is converted to Logic Graph nodes one token at a time, and the graph is interpreted directly — there is no separate parse-then-run step once it is in the graph. Scopes marked for a backend (`backend cranelift (...)`) are lowered before they run; everything else is interpreted, and the runtime may promote hot regions to compiled form. Compile-time evaluation is ordinary interpretation performed earlier: a `comptime` marker forces it where the guarantee matters (a result baked into the artifact rather than recomputed each run), but the common case is inferred — the compiler interprets the graph fragments a compiled scope depends on. Because compile-time evaluation runs under a capability set that excludes I/O (effects are capabilities; see *The logos strata and verification*), builds are reproducible by construction, while ordinary runtime code retains I/O. Parsing rules are Turing-complete: a logos's `constructor` is unrestricted code (surface forms as irregular as natural language are in scope), and a declarative consumption signature, where a logos chooses to carry one, is an optional fast path the recognizer can merge and index, never a ceiling. The generic prologue every `constructor` needs (bounds, this-token-is-me) is not written per logos: `logos`'s own `constructor` installs it, by reference to one shared checker, into every logos it constructs, so the contract is visible graph structure; core identities, hand-built by the seed rather than constructed through `logos`, are correct by construction. A `constructor` works against two states that stand at every source index: the *parsing tape* and the growing Logic Graph. The tape is the working frontier, holding both the synolons reduced so far but not yet final and the tokens still to consume, the latter lexed lazily on demand (reaching the next position is what pulls the next token from source), so lexing and parsing are one pass the constructors drive rather than two. A constructor reads forward on the tape for the tokens it consumes and back on the tape for its left context, the synolons reduced there so far. When precedence guarantees a cluster will consume no more tokens it is final, so it detaches from the tape and attaches at the current location in the graph, and the next locations are decided by the earlier synolons: an `if` opens a slot for a condition and a body and an optional `else` body, and stays incomplete until those are parsed and placed relative to it. The tape carries `insert`/`remove`, so a constructor may splice tokens in or drop upcoming ones before they lex, and that is the whole macro and custom-syntax mechanism: a syntactic abstraction is a constructor rewriting the token stream over real synolons, not a macro sublanguage. A constructor never hands a result to a scheduling driver: it edits the tape *in place* — replacing its own token with the synolon it built, or with another token, splicing out the neighbours it consumed, inserting tokens elsewhere on the frontier — and the driver decides only *when* constructors run (the precedence decision), never what they leave; what a constructor reports is just applied-or-declined, an explicit signal because a tape left holding a token is a legitimate outcome, not a refusal. The tape is therefore not a private parser structure but **Logic Graph with a string extension**: a reduced cell is a graph synolon already, and a token cell is a span of source text plus a candidate identity — a synolon-in-waiting — so reflecting on the tape needs no second metalanguage, and tape positions, tokens, `insert`, and `remove` enter as ordinary identities (the sketch's `parsing_tape`/`token` machinery) for the constructors self-hosting writes in Logos. *(Ruled in discussion, July 2026.)* The tape is disposable working state, so its splicing corrupts nothing durable; the source text and the graph it builds are what persist, and a re-parse rebuilds the tape from source.

**Elaboration is deferred-reduction operator precedence over the tape.** The driver needs only one token of lookahead: it lexes the next token, reads its `precedence`, and shifts or reduces the frontier accordingly, the ordinary operator-precedence decision. Precedence is an `f64` deliberately: fractional values let a new operator slot between any two existing ones without renumbering — higher binds tighter, associativity breaks ties. Reduction is deferred, so a token's identity is fixed not when it is first scanned but when it is reduced into its synolon, which is what keeps tokens mutable until then and makes a reduced synolon frozen against further token rewriting. That is what lets a higher-precedence operator to the right still rewrite the tokens on its left: it runs while they are still tokens (their reduction has not been forced yet) and they reduce afterward with their new identity, while anything already reduced to a synolon is immune. A constructor may look arbitrarily far forward where its own syntax needs it (lexed lazily) and may rewrite tokens behind it, but that backward reach is bounded to its own scope. Matched brackets are that bound: a bracket pair is a scope boundary that forces its contents to reduce into one synolon, caps a constructor's backward reach at the enclosing bracket, and is the point the token stream re-synchronizes on after a structural edit. Brackets are structural and always nest (`(` `)`, `{` `}`, `[` `]`, and any construct that opens a scope), the same nesting the scope spine records, whereas separators such as `,` are ordinary within-scope tokens, not boundaries. Token rewriting is therefore a within-scope, parse-time power (the Turing-complete parsing above, unchecked in the seed and admitted later only at a level that proves it terminating and deterministic); anything that must reach across scopes operates on the reduced graph through the rewriting engine, not on the token tape.

**Expressions are self-delimiting; `,` is the one explicit separator.** A scope's body is a sequence of expressions with no terminator: the finality rule above — a cluster detaches the moment precedence guarantees it will consume no more tokens — is also what ends each expression, so consecutive expressions simply follow one another, each complete in itself, and in most positions nothing at all stands between them. Newlines are ordinary whitespace and `;` does not exist; both are recorded as rejected, a terminator being syntax spent on a boundary the constructors already decide. Separation must be explicit only where adjacency would otherwise read as consumption — where a constructor whose trailing operand is *optional* sits before a value it must not take: `f(i32 3)` is one argument (juxtaposition: `i32` consumes `3` into an anonymous value with its logos), while `f(i32, 3)` is two, the logos itself and an uncommitted literal. The mechanism is precedence, not a special rule: `,` carries the higher precedence, so it is constructed before `i32` reaches for an operand, and a constructor that accepts not consuming its right-hand side then yields its own synolon as-is — the logos as a value. The same `,` therefore separates fields, arguments, and expressions, one separator everywhere; and like the optional `return` it may also be written where nothing is ambiguous at all, purely for the reader, marking the step boundary other languages spell `;`. *(Settled in discussion, July 2026.)*

**Hosted text does not owe a single reading.** Natural language is in scope as a surface form (the Turing-complete parsing above), and hosting it needs no new substrate: a hosted clause — Latin, English — parses to data, and data is a legitimate resting state. Per *Execution is function application*, a synolon whose logos is not a `fn` is never run, so the clause enters the graph as nodes and simply stands, open to the recognizer, reflection, and rewriting but to nothing that requires a single reading. Where recognition is ambiguous the synolon carries every identity that matched; where the *structure* is ambiguous it carries the alternative sub-synolons themselves, a parse forest held as plain value structure; where the system knows nothing at all, the faithful floor is the text literal (*Text literals are plain values*). No obligation attaches at parse time: resolution is lazy, at a use site, where later information either eliminates the alternatives or the demand surfaces `?`, the universal unknown — never `undefined`, which keeps its meaning of nothing-was-written. What hosted language is *for* is truth: a declarative sentence denotes a proposition — an ambiguous one, a set of candidates — and "executing" it is the proof layer applied to that proposition, the checker the `fn` and the proposition its operand, so the evaluation rule stands untouched. The kernel will rarely certify such a claim — proof establishes only what can be specified, never the truth of arbitrary claims (*Received code is checked before it runs*) — so an empirical claim enters as an asserted axiom with provenance, committing to a branch of *the ecumenical proof system*, where contradictory testimony coexists and the coherentist layer grades how sure: proven, refuted, plausible-to-a-degree, unknown, its rankings feeding what to try proving and never counting as proofs. Epistemic status is therefore assigned lazily and revisably by the proof layer, never by the parser: the same untouched sentence climbs from unknown toward plausible or refuted as the web around it grows. Recorded as rejected, to stay rejected: driver-level backtracking (stored branches, rollback of finalized synolons) — finality stays guarantee-based and one-way, exploration lives before commitment (in guards and in-constructor lookahead), and genuine ambiguity is represented, not searched; and `undefined` as the surfacing of ambiguity, which conflates too-many-readings with nothing-written. *(Settled in discussion, July 2026.)*

**Compilation is a reader of frozen structure.** A compiled artifact is a long-lived *reader* of a subgraph's structure; what licenses it is the absence of any structural writer over that region for the artifact's lifetime. So the rule that a `graph_mut` invalidates outstanding readers *is* deoptimization — a structural write to compiled code drops the artifact and falls back to interpretation — while ordinary runtime values flow through unaffected, because what froze is the code's structure, not its data. The interpret-versus-compile choice is then an adaptive matter of profitability, not semantics, and needs no syntax: unannotated code is interpreted while cold or unstable and compiled once it is hot and frozen, with `comptime` the one execution marker that is genuinely semantic (it fixes *when* evaluation happens and what is baked into the artifact). The seed, however, does not promote automatically: it interprets everything and compiles only what is explicitly directed — and v1's direction is the `fn` logos's own shared `compile` member (*Execution is function application*): write the function's body, call `f.compile()` in the pass, and the next call jumps to the installed code instead of walking the body *(settled in discussion, July 2026)*; the scope-granular `backend cranelift (...)` marker and profile-driven automatic promotion are later work riding the same deopt boundary. The same freeze licenses optimization — equality saturation needs a fixed input — so compiling a region, optimizing it, and extracting a cheaper proven-equivalent variant are one operation under one condition: no live structural writer. Editing itself runs only this analysis machinery, which is pure (it holds no I/O capability) over the changing graph; the user's program runs nothing until invoked, and the language's own machinery, being hot and structurally stable, sits permanently compiled and untouched by edits to user code.

**Mutable code is compilable; the reader–writer rule, not mutability, gates compilation.** A region classified `mut` is not barred from compilation — only a region with a *continuously live* structural writer is, and then only because the next write would deoptimize it, so compiling it would not pay. A mutable region that is currently stable compiles and runs at native speed like any other; the "`mut` stays interpreted" rule is this profitability fact, not a prohibition. What governs compilation is the one borrow rule applied to code as data: a compiled artifact is a *read reference* to code structure, so a structural write needs exclusive access and therefore drops every such reader first — that drop *is* the deopt. Two readers of code must not be conflated, because different timings keep them fresh: the **name index** (the resident trie of `id_context`s) never mis-resolves, because resolution filters candidates by open-scope membership: each `id_context` carries the scope it was declared in, and one whose scope is not live is skipped, so a stale entry is inert rather than wrong, and removal is cleanup on explicit source mutation rather than a per-edit correctness step; **`bcode`**, by contrast, `run` trusts and does not re-verify per call, so its freshness is the deopt layer's job (null the stale `bcode`, fall back to the body-walk), not the writer's. Until the borrow checker makes "no structural write while a code reader is live" static, this exclusion is a discipline the program keeps, exactly as with every other mutation.
**The callable ground is `@exec`: a jump capability, minted rather than checked.** *(Settled in discussion, July 2026; supersedes an interim `bcode {entry, arity}` wrapper, rejected as packing the caller's knowledge into the callee's pointer.)* Beneath `fn` there is exactly one executable primitive: `@exec`, an address the program may jump to — `fn`'s `bcode` field carries one inside its `callable` value (the entry paired with its declared convention), and nothing more. Executability is a property of pages, not bytes (the NX bit), and no instruction can ask "may I jump here?" — the only answer is the fault — so Logos never checks it: an `@exec` value can only be *minted* by the machinery that made the memory executable (a backend finalizing compiled code, or an explicit foreign declaration), the same fail-closed shape as every other capability. A jump that lands on non-executable memory is an upstream invariant violation, not a condition to test for per call. The *calling convention* — which registers carry which operands on which platform — is deliberately no part of `@exec`: it lives in the two pieces of code on either side of the jump, both emitted by a backend from the same abstract signature (the `input` record's order and widths), so platform variation is invisible wherever one compiler emits both sides; and the signature survives sealing (a seal keeps what has been read out — logos — while dropping the body), so nothing must ride inside the callable. Conventions are declared metadata — ordinary identities a backend renders per target, open-ended rather than an enum — and they carry decisive weight only at the FFI boundary, where the foreign side froze its convention and the declaration is what lines the registers up. Boundary overhead follows: interpretation pays a trampoline per call; a compiler that sees both sides may inline, re-register, or delete the call entirely (co-compilation, including the JIT promoting hot regions); only genuinely foreign code keeps a boundary no optimization removes. Concrete machine operations are identities — `add_i32`, `fadd_f64` — each immutably carrying its `@exec`; a surface operator like `+` is the parse-time constructor that *resolves* an application to one of them, storing the chosen operation in the synolon, so nothing mutates and nothing is looked up in a side table: to evaluate a synolon, consult its logos; if the logos carries code, jump. Alternative implementations then live where versions live — versioned scopes — not in per-phase tables. The seed's run tier now lives this paragraph (July 2026): every native is a `callable` value — `[entry: @exec, convention]`, the logos of `add_i32` and of a compiled fn's code alike — referenced from the synolon's own op slot, and the interpreter consults no table anywhere; the Cranelift lowering still rides a Rust-side table until lowering re-keys per backend identity.

**Foreign code is a declaration, and the C ABI is the lingua franca.** *(Settled in discussion, July 2026.)* An `extern` declaration names a library, a symbol, a signature, and a convention; the loader resolves it and mints the identity's `@exec` — minting by foreign declaration, the second of the two licensed mints — and an unresolvable symbol fails at load, never as a wild jump. Interpreted callers reach foreign code through a minted *thunk* (the backend compiles one adapter from the interpreter's convention to the foreign one per declaration), so the interpreter stays convention-blind; compiled callers emit against the declared convention directly; and a Logos function crosses the other way by being compiled against the foreign convention and handing over its `@exec` — callbacks are the same mechanism reversed. Data crosses on the platform's terms: struct layout is C-compatible (natural alignment and padding — diverging from the platform ABI buys nothing), and text bridges explicitly (`«…»` is `[len][bytes]`, not a NUL-terminated `char*`). C is the whole interop strategy: Rust exports over the C ABI (it has no stable one of its own) and C++ is taken through `extern "C"` surfaces, so winning C wins both transitively — essential while the library ecosystem is young. Header ingestion is the seamlessness multiplier, and it ends as the parsing doctrine applied to `.h` files: a constructor that consumes C headers and emits declarations, mechanical bindings no human writes.

**Execution is function application, and only functions execute.** Everything runnable is a `fn`; operators (`+`, `=`), field access, and even control flow are functions, so evaluation has one rule: to evaluate a synolon, read its `logos` — if the logos is a `fn`, invoke it on the synolon's hyle (its operand record); otherwise the synolon is data, read through its logos's layout. A synolon whose logos is a function is therefore an application, exactly as the operand-record model already implies. Control flow works as a function precisely because operands arrive *unevaluated* — a function receives its operand record as Logic Graph and decides what to evaluate, so `if` runs its condition and then only the taken branch; laziness is the default the substrate gives, not a special form. The `fn` logos carries two shared functions: `compile` (lower the body to machine code, its `bcode`) and `run` (execute the function), plus per-instance `input`, `output`, `body` (the reflectable Logic Graph), `bcode` (the opaque compiled code, null until compiled), and `frame` (the activation-record byte size the function's parameters and locals occupy per call, null when it has none). `run` is one primitive with two paths: if `bcode` is present it jumps to it; if `bcode` is null it walks `body` directly. Interpretation is therefore not a separate function but `run`'s null-`bcode` path, which is why deoptimizing is just nulling the stale `bcode`: the next `run` falls back to walking the body, fail-closed by construction. `run` does not re-verify *whether `bcode` still matches `body`*; that freshness is the checker and deopt layer's job, not paid on every call. One reflection consequence follows: only the body-walk path materializes a reflectable live stack, while a compiled `run` is an opaque leaf (its frames live on the machine stack), so introspecting a running function means deoptimizing it first. The two are simple, predictable primitives; the policy that chooses between them is not theirs.

**A function's surface is `fn (params) -> logos (body)`, and its parameter list is a logos.** A function literal is `fn ( <params> ) -> <return logos> ( <body> )`, an anonymous value that a name binds with `:=`, as in `main := fn () -> i32 ( return 40 + 2 )`. The parameter list *is* a record logos, its fields written `name : logos` or as a bare `name` that accepts any synolon, with an empty `()` the empty record; a function's `input` is therefore literally the logos the parentheses define, and building a parameter list is the same work as building a record logos. The return logos after `->` is mandatory, `-> void` when there is none. The body is a matched-bracket `( )` scope whose value is what it evaluates to, its trailing expression; an explicit `return` is optional and only exits the function early. Brackets divide by role: `( )` delimits scopes, parameter lists, and bodies; `[ ]` is reserved for indexing; `{ }` is unassigned.

**A scope's value is what it evaluates to, and `return` is an optional early exit from the enclosing function.** Every `( )` is a value-producing scope, and its value is whatever its body evaluates to: its trailing expression. So `a := ( sum := i32 0  ...  sum )` gives `a` the value `sum`, and `fn () -> i32 ( x + 1 )` returns `x + 1`, with no keyword in the common case. `return` is never needed to produce a value; it is written only to leave a function early. `return X` unwinds to the nearest enclosing *function* and makes `X` its result, passing out through any loop, conditional, or block in between, the ordinary early return. A `return` inside a loop inside a function therefore leaves the function, not the loop; a value that is only a block's result is written as that block's trailing expression, not a `return`. A `-> void` function, or a scope whose trailing form yields nothing, has no value to carry. Whether a given `( )`'s value is taken or discarded is still fixed structurally by the construct that introduced it (a bound block's value is used, a loop body's is thrown away): that governs where a value goes, not how it is produced.

**Operands travel on the stack, which is why `run` and `compile` take no arguments.** Each reads its own fields (body, signature, code); the call's operands are placed on the stack by the caller and read by the callee's code — the ordinary calling convention. Under interpretation this copies each operand from its synolon onto the stack per execution (the interpreter's standing cost, as in any bytecode VM); under compilation the synolon is read once at compile time and operand access is baked into the machine code — immediates inlined, locals assigned to registers or stack slots — so compiled code touches no synolon at run time and runs at native speed. Because operands live on the per-thread stack and never in the shared graph, execution is concurrency-safe by construction: the shared Logic Graph is immutable code, the mutable per-call state is each thread's own stack frame. This is also what makes recursion work — each activation gets its own frame — independently of, and prior to, any borrow checker.

**A logos's metadata is shared by its values; its per-value data is not.** A logos stores its `precedence`, `associativity`, `constructor`, and `destructor` once and shares them across every value of that logos (its `shared` members); the bytes that differ per value are the per-instance fields. The record case — a logos whose `constructor` derives the layout automatically, reading the field declarations in its scope and filling `fields` and `size_bytes`, so defining ordinary data needs no hand-written layout — and the authored case, where parsing or semantics are written by hand (operators, primitives), are **one identity, not two**: the former `struct`/`type` pair was "nearly identical by design, differing only in that a struct auto-derives what a type leaves to its author," and the merge *(ruled July 2026)* makes the derived layout a default of `logos` rather than a second identity. Field *names* are not stored on the record: they enter the shared name index (the trie) like every other declaration and resolve by open-scope filtering (see *Name resolution is scope-filtered*) — a per-record `names` hashtable was considered and rejected, because resolution is one mechanism and a second per-logos name store would shadow it. The `shared` keyword itself is a placeholder, not surface syntax to keep: placement is *derived* from mutability (see *Two muts, and the storage partition* — a never-`mut` member is identical across instances and stored once with the logos by decision, not annotation), so when `mut` and `pub` land, `shared` disappears and reflection reports a member as shared by reading that it is immutable. *(Settled in discussion, July 2026.)* The seed stores this head in full as each identity's record, and **the constructors drive the tape**: the schedule byte that stood in until they did is retired *(stood down, July 2026 — the seed's driver is the minimal deferred-reduction loop, and each identity's `constructor` does its own consumption)*. The driver classifies a token from the record it already reads: the `precedence` field doubles as the extender signal — NaN (never extends an expression to its left), finite (an infix operator, shifted and reduced), +infinity (a tight extender — the call `(`, postfix `.`/`@` — its constructor invoked immediately over `tape[-1]`) — combined with constructor presence and the record kind (a bare delimiter token against a data logos). A *declarative consumption signature*, where a logos chooses to carry one, remains the optional fast path *Source becomes runnable* licenses — never a ceiling, and no longer a stored byte. The `constructor` is a `native` callable leaf (invoked, never read into, per the reflection boundary) under one entry signature that self-hosting replaces with Logos source; and the `destructor` is the undefined null until the seed implements the ruled drop model — `drop x` runs `x`'s destructor, constructor-inserted `defer` teardown, no implicit scope-end destruction (see *Memory and concurrency*).

**Build and run are one self-directing pass.** Lexing, parsing, and execution interleave: a construct can `compile` and then `run` a function during the same pass, immediately, rather than deferring it. There is no separate promotion scheduler in the seed — execution is directed explicitly where it is wanted, and adaptive hot/cold promotion is a later layer over the same boundary. A `run` over a scope that compiles and executes on the spot is therefore comptime metaprogramming in full generality — the whole language available at parse time, operating on real Logic Graph rather than a macro sublanguage — and it stays reproducible because comptime runs without the I/O capability, the same boundary that separates compile-time from run-time effects.

**A logos is a comptime value, resolved in the pass.** A function may return `logos`; a call to it is evaluated at comptime, during the one pass, and the concrete logos it produces substitutes for the call — a dependent declaration `a : metalogos(0)` is the ordinary declaration mechanism over a computed logos (chosen over runtime logos-values: logos are comptime). `.logos` is the reflection metaproperty reading any synolon's logos; `==` and `!=` over logos values compare by identity, sound because logos identities are interned; and an `if` whose condition is comptime-known folds at parse, its untaken branch dropped unparsed — which is what lets branches for *other* comptime logos coexist in one source. *(Settled in discussion, July 2026.)*

**A program is a file, and the binary stays out of the way.** `logos file.logos` runs the file top to bottom — there is no `main` entry point: the top level *is* the program, and a trailing value prints (a stand-in until I/O lands). Bare `logos` is the REPL over the same one pass, and a failed line rolls its declarations back out of the name index, leaving no trace. The binary has no subcommands and no compile flags, ever: what compiles, links, and builds is decided *inside* Logos source (build and run are one pass), never on the command line — `--help` is the only flag. *(Settled, July 2026.)*

**Two graphs, one source of truth.** The *source graph* is file-backed, persistent, and the program's identity; a build is a pure function of it. The *runtime graph* is generated during execution, ephemeral, and owned exactly like other runtime data — arena-allocated (arenas already carry the cyclic structures the graph needs), with the same lifetimes and no garbage collector. Runtime-generated structure is reflected upon the same way source is: by structural traversal, discovered by reachability from the use sites that reference it; there is no value-level or time-travel reflection (a value to inspect is stored in a variable, as in any language; reconstructing past states is the debugger's role over DAP). Reproducibility needs no separate leveling: a build reads only the files, and compile-time evaluation runs without the I/O capability, so the artifact is a pure function of the source. Cleanup needs none either: derived and runtime structure share the lifetimes of the scopes that create them, so when a scope ends its generated graph is freed like any other data — *reclamation*, the effect-free return of memory, which is the one thing that stays implicit precisely because it runs no teardown; anything with real teardown carries constructor-inserted `defer` structure (see *Memory and concurrency*) — and when the program ends everything generated is gone. The runtime may modify the source, but only on explicit command: the change is a *promotion* — its dependencies are closed over, it must be free of runtime-only inputs (or have them reified as constants), and it is copied out of its arena, unparsed to a source diff, re-checked, and committed under gate. Source is never silently rewritten, and converting any subgraph back to source text (unparsing) is a presentation and serialization facility, not the reflection mechanism.

**Versioned scopes.** A scope may have more than one implementation: the canonical one written in the source, plus any number of derived alternatives — an optimized rewrite, a specialization for known arguments, a different backend — that are proven equivalent and chosen by cost. These alternatives are the structural form of the rewriting engine's equality classes: registered at the scope's entry, extracted by a cost function. No global level numbering or edge classification governs them, because none is needed. A running program may attach optimized variants to a function's live representation and discard them when the enclosing scope ends, exactly as it discards any other runtime data; the source file is untouched unless a variant is explicitly promoted. What gets cleaned up is bounded by lifetimes; what is reproducible is bounded by the files being canonical. (This is unrelated to the proof layer's *universe levels* — the Lean-style hierarchy that prevents self-referential paradox when proofs quantify over proofs; the shared word names different mechanisms.)

**Backends are identities, and lowering is keyed by them.** *(Settled in discussion, July 2026.)* A backend is an ordinary identity carrying its own mapping from node kinds to emission code — its rule set lives with it, so adding a backend is adding an identity and its rules, never a core edit — and multiple backends coexist by construction: a scope's compiled forms are versioned-scope alternatives ("a different backend" is already one of the listed kinds above), extracted by cost. Nothing per-identity may hard-code a single backend — one lowering slot per identity would repeat the mistake of one mutable code field on `+` — so the lowering dimension is per backend, per identity. `run` stays backend-neutral because a compiled form is an address plus a declared convention: whoever minted it, the call is the same; code from different backends interoperates through shared conventions exactly as foreign code does, with co-compilation erasure available only within one backend's region. A backend also declares the *gate* predicate over the scopes it accepts — what it rejects simply stays on other versions or stays interpreted, fail-closed. Non-CPU backends (hardware synthesis) fit the same structure with two honest stretches, deferred with it: their artifact is not an `@exec` (the callable is a CPU-side driver stub, the artifact riding the version as data), and their call semantics want asynchrony, which is the effect system's surface, not the ABI's.

**Concurrent access.** Reading shared graph structure across threads uses ordinary shared borrows (`&`): many threads may read the standard library, definitions, and any subgraph at once. Writes are either disjoint — the `parallel for` pattern the borrow checker already recognizes — or exclusive (`&mut`); concurrent mutation of the same synolon is rejected at compile time, as elsewhere. Per-node reference counts are deliberately avoided: they leak on the graph's cycles and track liveness rather than the access exclusion threads need; arena ownership governs lifetime instead. The one genuinely shared, mutable structure is the identity interner — allocating or looking up an identity from multiple threads must be synchronized — together with any equality graph that parallel rewriting builds.

## Identity recognition

How the graph answers "what is this?" — for parsing, for reflection, and for rewrite matching — with a single engine. *(Target architecture; the bootstrap ships the trivial version, detailed at the end of the section.)*

**One engine serves parsing, reflection, and rewriting.** Pointed at a node, the recognizer reads the surrounding structure and returns the identities that match there — the operator a token sequence denotes, the construct a subtree forms, the rule whose left-hand side fits. This is tree/term pattern matching — the structural analogue of regular expressions over text, extended with e-matching where the match is over equality classes rather than syntax — and structural grouping, reflective query, and rewrite-rule matching are clients of the one mechanism. (Recognizing tokens from source *text* is a separate, string-keyed layer — a trie over names — not this system; the two are different ways of reaching the same identity, by name and by structure.) Matching is boolean and exhaustive — a node either has an identity or it does not, and the default query returns *every* identity that matches; there is no fuzzy or probabilistic matching in the core. Where a single answer is required (which parse wins), a *deterministic* precedence/cost order selects it — an order, never a statistic.

**Meta-navigation walks the graph; the scope stack is the graph's own spine.** The structure navigation walks is the one elaboration builds, the Logic Graph itself, and the parse-time *scope stack* is not a separate structure but the chain of enclosing scopes from the current location up to the root, read off the graph: entering a scope is descending into a subtree, leaving it is ascending, and name resolution is the walk up that spine (accelerated by an O(1) open-scope membership set where the batch pass does not already carry the environment). The *parsing tape* is the orthogonal axis, the working frontier of a scope (its not-yet-final synolons and pending tokens), which the constructors consume and turn into graph; a matched bracket is where the two axes meet, since closing a scope reduces its whole tape frontier into one synolon on the spine. So the two axes are distinct, the live per-scope frontier on the tape and settled structure behind it on the graph spine, and the same graph walk a `constructor` uses to place and reach its operands is the walk reflection and rewriting use over built structure. That is the concrete substrate under the one recognizer above. *(Architecture reached in discussion, July 2026; the surface syntax for general meta-access is still open.)*

**Recognizers compile into one shared, stratified structure.** All recognizers merge into one graph; run over a node, it yields the set of every identity that matches there, computing shared sub-tests once rather than re-checking each recognizer separately. The structure is stratified by tractability: a fast *observable skeleton* — navigations and discrete tests on identity, logos, arity, and existence — is merged, minimized, and indexed; structural repetition and nesting become cycles and a stack; and arbitrary computation is confined to *guards* at the frontier, run only on the few candidates the skeleton has already narrowed. The skeleton carries the speed, the guards the expressivity. The optimized paths are straight-line — forward dispatch and fail-fast, no backtracking; a condition that compares two nodes is a guard, not a dispatch key, and a relational or negative condition (two operands differ, say) cannot prune the skeleton at all — it rides along as a residual filter or is served by an auxiliary index, exactly as a non-sargable predicate is in a database.

**The structure is itself Logic Graph code — compiled yet extensible without recompilation.** The merged recognizer is not a trie or automaton walked by a special machine; it is ordinary Logic Graph code, run over the start node like anything else, so the same adaptive machinery interprets it cold and compiles it hot. Each identity's recognition code is first minimized and canonicalized — by the ordinary rewriting engine under a compute-cost function, since recognizers are themselves Logic Graph code — so that recognizers branch the same direction wherever they can; equivalent ones then converge onto the same shared paths, and all of them merge into the single graph. Adding an identity is a local merge that reuses those shared paths, hash-consed like any other node, not a global rebuild; the merged graph is allowed to drift from minimal and is re-minimized as a periodic recompaction rather than on every edit. The convergence is guaranteed on the decidable skeleton and best-effort above it, where program equivalence is undecidable and only identical guards merge.

**Admission is a proof obligation, and identification is itself proof.** A candidate recognizer is admitted at a recorded *level* — proven safe (terminating, deterministic, effect-free) or, during bootstrap, not-yet-refuted — and a use-site demands the level it needs, so safety is paid for where required and relaxed where bootstrapping. What is refuted, or resists checking where a use-site demands proof, is rejected; there is no level below not-yet-refuted and no escape past a use-site's demand. This is the proof layer wearing another hat: "X is identity I" is the proposition "X has I's defining properties," so a trivial pattern is a trivial proof and a rich property — totality, logos-correctness, a borrow that never conflicts — is a real one; identification, logos-checking, borrow-checking, and termination-checking are one activity at different strengths. The system runs that activity on ordinary code, on candidate extensions to itself, and on itself, bottoming out — as everything does — in the small hand-trusted seed it cannot prove.

**Queries specialize the structure.** Beyond the default all-match, a directed query ("is this matrix-addition?") prunes by intersecting the question with the saved recognizers — an automaton product, or in its general form a partial evaluation that bakes in the query's constants — yielding a specialized recognizer that explores only the relevant paths; computing that intersection is itself a job for the identifier. The first pass over freshly tokenized input is discovery (all-match, building structure); directed queries are interrogation over structure that already exists.

**Identification is incremental.** The durable state is the source text, the resident name index (the trie of `id_context`s), and the graph; there is no persistent token array, since the parsing tape is disposable working state rebuilt from source. Re-identification follows the reader–writer rule: each cached identity is a reader of the source span and rules it consulted, and a write invalidates exactly its readers. Most edits are local; the two that are not are handled explicitly. Redefining an *identity* ripples to its uses, found without a full scan through per-subtree *use-sets* (a summary of which identities occur below each node, carried as one more component of the bottom-up state). A structural edit (an unclosed delimiter) is handled by re-lexing forward from the edit to the point where the stream re-synchronizes. The seed and runtime self-modification take the coarse path instead: only the graph is mutated in place, and editing the source *text* simply re-parses it, which is rare because a program rewrites its own *graph* for iterative self-improvement, not its text.

**This is the target architecture, not the seed.** The bootstrap ships the trivial version — recognizers run directly and unmerged, guards interpreted — and grows toward the fully merged, minimized graph, e-matching, and specialization as measurement on real graphs warrants, never before. The shape it grows into recurs throughout the system: a fast, decidable, canonical core; escape-hatch tiers for the general case; deterministic disambiguation; and compilation maintained incrementally with periodic recompaction — the same architecture as the borrow checker, the compiler, and the rewriting engine, which is why one set of mechanisms serves them all.

## Identities, names, and metadata

How identities are stored, named, and annotated: the node store, the optional name index, and how lifetimes and permissions attach without bloating every node. *(These are design decisions not yet validated against a real corpus; the bootstrap can use the trivial version of each and tighten later.)*

**The store is keyed by address.** An identity is an interned handle that lives in some backing store — an array, an arena, or any allocator that hands out stable addresses — in which the identity's *address is its id*. Because every node is a fixed two-pointer (16-byte) cell, that address is literally an index; the store needs no separate key and no hash: the id *is* the address. A content → id hash is a *separate, optional* index, needed only where structural **sharing** is wanted — interned canonical identities (one `i32` referenced everywhere) and the e-graph. Ordinary source nodes are not deduped: each occurrence of `a + a` is its own node with its own address. Identities are *per-run*: addresses, and hence ids, are assigned at runtime and are never persisted. The durable form of any graph is source text — files are canonical, promotion serializes by unparsing — and the graph is rebuilt at load, with disposable content-keyed caches as accelerators where startup cost warrants. Anything that must survive a session (caches, compiled artifacts, certificates) keys by content or structure, never by id.

**Names are an optional index, not the management mechanism.** Every identity — named or not — is reached by its handle and by structural traversal from its uses, so the handle is what you compare, store, and refer to. A name is a second index (string → handle) for the subset that source refers to by word — operators, logos, declarations, keywords. Most identities are anonymous subexpressions and carry none; naming them all would mean gensym-ing millions of meaningless names, which is bloat, not management.

**Name resolution is scope-filtered, and shadowing is disallowed.** The name index maps a spelling to a list of `id_context`s, each pairing an identity with the scope it was declared in. A use resolves by keeping the candidates whose scope is currently open (an ancestor on the *scope stack*) and taking the one that remains. Because a name may not be redeclared while another declaration of it is live (no shadowing, enforced by a single check at declaration time), exactly one candidate survives, or none: a genuine out-of-scope use, which still lexes as the name and then fails resolution with a precise error rather than an unknown-token one. Two survivors is impossible under that rule, so it doubles as a cheap internal invariant: if resolution ever finds two in the same scope stack, the index is corrupt. The ancestor walk runs over the graph's own scope spine, accelerated by an O(1) open-scope membership set during elaboration; the `id_context` list is resident for the whole run, so a spelling can be resolved later from any context by the same membership test, and entries are removed only when their declaration is deleted from the source.

**Metadata has three homes, chosen by frequency and origin; the model is uniform, the storage is not.** *(Revised June 2026; supersedes the earlier wrapping design.)* A wrapper around a node would make every read of its logos yield the wrapper, and a wrapper spliced into the hyle chain would falsify the prime invariant that the logos defines how the hyle is read — so no facet wraps. Instead a facet lives where it is naturally read: **(1) universal and tiny** — the construction phase, the four bits every node passes through — rides in the node's own tag bits (see *Mutability and construction*), read on every access because the node is already in hand. **(2) Syntactic deviations** — what the user writes on a declaration: `pub`, per-field `mut`, capability and backend modifiers — are read off the declaration's structure at resolution time, the same lookup `.` already performs, so they are not side-data at all. **(3) Inferred deviations** — facts analysis derives that are *not* in the source: a lifetime extended past its scope by escape, a borrow state — are reflectable nodes reached through the derived id → metadata index below. Intrinsic inferred facts (a non-default lifetime) share one per-node metadata record found by that index; relational and event facts are independently-reachable nodes by nature — a borrow references its target, a move relocates a scope-level drop node — so they need no index entry on the target. The giant common case deviates in none of the three: scope-tree defaults supply lifetime (the enclosing scope) and visibility (own scope and descendants) with nothing stored, the phase bits clear at freeze, and the node stays sixteen bytes. The asymmetry is the rule — *a deviation the user writes lives in the declaration; a deviation analysis infers lives in a fact-node* — and a facet that can be either, such as an annotated versus an inferred lifetime, enters by whichever door it came from. (Source positions are not a node facet at all: a node's position is the source span it was parsed from, held in a derived source-map that is one-to-one with source, rebuilt from source, and absent for runtime-generated nodes; fine-grained permissions are carried by the reference, per *Reference-granular permissions*.)

**Lookup rides the scoped walk; the index is only an accelerator.** During checking, lexical scoping guarantees a binding's declaration is an *ancestor* of every use, so a top-down walk carrying its environment (push bindings on scope entry, pop on exit — standard scoped-checker behavior) always has the binding's metadata in hand at a use; no index is needed for the batch pass. Cross-references that skip ancestors — to logos, functions, definitions elsewhere — are exactly the `'static` / default cases with no non-trivial lifetime to look up. A derived index (id → metadata node) earns its keep only for out-of-walk access: re-checking a single node after an edit, or a tooling hover, where there is no environment built up.

**A uniform model is not uniform storage.** Every identity *has* a permission and a lifetime in the model, but almost none *store* one. The node stays lean — just its logos and hyle (16 bytes), the handle being its address rather than a stored field; metadata hangs off only on deviation; defaults are inherited from the scope tree (permissions on scopes, lifetime from the enclosing scope, dropped at its end); and the giant common case — immutable `'static` definitions: operators, logos, the standard library — shares one default of readable-by-all, with no lifetime tracking, no drop node, no borrow state. Storage is proportional to *deviations*, not to identity count, which is what makes a uniform model affordable.

**A declaration has a stable identity across edits, distinct from its content hash.** The per-run address (*The store is keyed by address*) is ephemeral, and the optional content hash changes on every edit, so neither is a stable handle on *the same declaration* as its value changes under an editor. That handle is nominal: the pair of the spelling it was declared under (optionally a regex, for pattern-recognized identities such as numeric literals) and the declaration context that distinguishes same-spelled identities. It is stable under edits to the value and changes only under rename, the opposite stability from the content hash and complementary to it: a cache keys by content and should miss when content changes, whereas a proof in progress, a cache of derived facts, or a collaborator's cursor must follow the same declaration across content changes and therefore keys by the nominal identity. It is device-agnostic, since two systems sharing a root context resolve it identically, which is what lets a cross-device reference and a re-parse both re-associate persistent metadata against the right node. Like the name index and the id → metadata index, it is a *derived accelerator*, not a field on the node, so it stays off the hot path: interpretation dereferences the address directly, and the nominal identity is consulted only for serialization, cross-device exchange, and re-parse correspondence. It covers only *named* declarations; anonymous subexpressions (*Names are an optional index*) are re-associated structurally or by content. For a CRDT over the graph itself (*The IDE as platform, and the reflective web*), the nominal pair is only the human-stable half of a node id: a globally unique, origin-attributed component (a replica id and a logical counter) is added to it, to distinguish independently-created same-spelled declarations and to give anonymous nodes an id at all. *(Design direction reached in discussion, July 2026.)*

## Standard library and ecosystem strategy

The standard library is built **before** release, avoiding the fragmentation that plagues languages shipping thin stdlibs. By the time Logos is available, the standard ways to do common things are decided, documented, idiomatic, and tested by real programs.

It includes: collections and the iterator protocol `for` desugars onto; Unicode-aware strings and parsing; numerics including arbitrary precision and the cost functions for stability-aware rewriting; symbolic math (expressions, algebraic rule sets, differentiation, integration, polynomial and linear algebra, domain support, verified rules where possible); allocators and smart pointers; concurrency and async primitives; I/O, networking, and filesystem (files and folders as Logic Graph values); verification utilities (SMT integration, refinement predicates, inductive logos, common tactics); the compiler-extension framework; and the documentation generator.

The cultural commitment matches the technical one: **single canonical solutions, opinions formed, batteries included, idioms documented.** Competing libraries exist, but the standard library defines what idiomatic Logos looks like, and the language's design is opinionated enough to back it.

## Tooling philosophy

- **LSP-first.** A Logos-written LSP server delivers highlighting, errors, autocomplete, go-to-definition, hover, and refactoring to any LSP editor, so users keep the editors they already have. Because so much information is already in the Logic Graph, the server is richer than equivalents in other languages.
- **Standard-protocol debugging and profiling.** DAP, standard samplers, flamegraphs — users keep the tools they know.
- **A Logos-native structural editor and IDE as long-term work.** Operating directly on Logic Graphs: semantic rather than textual editing, principled rather than heuristic refactoring, the editor itself customizable in Logos and able to ask "update all related occurrences?" with confidence. The effort is real but far smaller than equivalent IDEs elsewhere, because the meta-reflection does the heavy lifting that would otherwise require separate AST, analyzer, refactoring, and language-server stacks. Not on the critical path — LSP covers the immediate need.
- **Documentation generation as a flagship feature.** With logos, signatures, doc comments, examples, capability declarations, and proofs all in the graph, generated docs are richer than conventional tools — verified examples, proof obligations, capability summaries — and the generator is customizable Logos.

## The IDE as platform, and the reflective web

*(Design direction reached in discussion, July 2026; far-horizon, dependent on the base language, the compiler, and the structural editor landing first, and not on the critical path.)*

The IDE is Logos programs over the Logic Graph (*Meta-reflection and the unified system*). Because it is, and because Logos has I/O and compiles to portable backends, the same reflective structure that is the editor can grow, without a rewrite, into a shareable, networked, rendering platform: a browser reached from the tooling end rather than built as one. This section records that direction and the properties the substrate already provides for it.

**The graph is edited as text or as graph, because the graph is primary.** A use is a reference to the identity it uses, an edge in the graph, not a spelling to be re-resolved on the fly (*Representation, elaboration, and lifetimes*). So text and graph are two renderings of one structure rather than two artifacts kept in sync, and the editor projects any region either way, as glyphs or as nodes and edges, toggling per region. Per-region is what keeps the graph view legible: a whole-program node-link view is an unreadable hairball, a focused region is not, and the arrows are the identity references already in the graph rather than a dependency view recomputed and re-synchronized on every edit. This is why the projection is cheap here and would be a synchronization problem in a text language, where the edges must first be recovered by parsing and analysis.

**Received code is checked before it runs, because the substrate carries its own proofs and capabilities.** A shared unit of Logos (code, data, or interface) can carry checkable proofs of its own behavior, in the manner of proof-carrying code: that it does not touch the network, that it reads only what it is handed, that it terminates. The receiver's kernel checks the proof cheaply before running anything, and the unit runs under the capability model with no ambient authority (*Memory and concurrency*), so it can touch only what it is explicitly granted. This is the design answer to the web's structural gap, that received code is trusted by default and inspectable never. Its reach is exact: it provides verifiable code *behavior* and verifiable *provenance* (which axioms or sources a result rests on, carried as the proof's branch, *The ecumenical proof system*), not the truth of arbitrary claims, since a proof establishes only what can be specified; and it applies to content built in the system, since legacy content carries no proofs to check.

**Reaching the web is compilation to WASM, not interpretation inside it.** The backend interface already lists WASM (*The architecture*). The platform compiles Logos to it and adopts the language-neutral substrate primitives: the WebAssembly component model and WASI for module interop, WebGPU for compute and for rendering the interface directly, QUIC with HTTP/3 and WebTransport for transport, and OPFS for local persistence, while rendering its own interface rather than reimplementing the DOM and the accreted web-API surface. Shipping the *interpreter* into WASM would stack the graph-walker's cost on WASM's own, and the interpreter's hot operations (pointer-chasing the graph) are what WASM handles least well; compiling sidesteps both. The interpreter's absolute speed is not on this path in any case: it runs cold and structurally-mutable code, and hot regions compile (*Source becomes runnable by interpretation*).

**Collaboration is real-time because source is the truth and identities are stable.** Source text is canonical and the graph is rebuilt per device with its own per-run addresses (*Two graphs, one source of truth*), so two systems collaborate by exchanging *source*, not addresses, and each resolves it locally. Real-time collaboration is then a conflict-free replicated document, persisted locally in OPFS and synced over WebTransport, at either of two levels. The near-term level is a sequence CRDT over the source text with each device re-parsing, which needs no graph-level identity and reuses established libraries; the nominal identity (*A declaration has a stable identity across edits*) and incremental re-identification (*Identification is incremental*) then preserve per-node metadata across each re-parse. The harder, later level is a CRDT over the graph structure itself, for collaborative structural self-modification, which needs every node to carry the origin-attributed identity described in that same paragraph and is complicated by the graph's cycles. Promotion (*Two graphs, one source of truth*: unparse a graph edit to a source diff, re-check, commit under gate) is the bridge that keeps even graph-level edits anchored to canonical source.

**The image's failure mode is designed out.** Smalltalk proved a live, reflective system can be productive and still lose on the practical properties a team needs: diffable, mergeable, collaboratable, deployable, interoperable. The cause was the image, a live mutable binary object graph held as the source of truth, which has no stable, comparable, portable form. Logos avoids each failure at its root. Source text is canonical, so diff, merge, and collaboration ride on text and on CRDTs keyed to stable identity, rather than on comparing heaps. Sealing (*Reflectability is a droppable capability*) collapses a region to a bare native artifact, the small deployable Smalltalk could not produce. Foreign code as a declaration and compilation to standard backends (*Native interop and the Rust bridge*) keep the system from becoming an island. The one residual risk is cultural rather than architectural: an everything-in-the-graph design must not harden into an everything-must-enter-Logos ethos, which is how the ancestor became a walled garden.

## Englogos: a Logos-native human language

*(Design direction reached in discussion, July 2026; far-horizon, dependent on the proof layer and the ecumenical system, and not on the critical path. Recorded as a direction, not a designed artifact: no vocabulary, morphology, or surface grammar exists yet.)*

**Englogos is the inverse of hosting: a human language built on Logos rather than an existing one taken in.** *Hosted text does not owe a single reading* accepts natural language as it stands — irregular, ambiguous — and represents what cannot be resolved. Englogos runs the other direction: a constructed, human-usable surface whose grammar is authored as Logic Graph constructors from the start, regular by construction, its complete rule set queryable at any moment. It needs no new substrate — it is the parsing doctrine ("surface forms as irregular as natural language are in scope") applied to a surface designed to be regular.

**The design target is Humboldt's property, engineered: finite means, exception-free, dense.** Human languages already make infinite use of finite means, but their rules are implicit, unenumerable, and shot through with exceptions that persist because enforcement is only social. The substrate's discipline is the opposite — the refrain across this document is one rule, no exceptions (one evaluation rule, one reader–writer rule, one resolution rule, one synolon) — and Englogos extends that discipline to a human surface. Its information density comes from the generality of each rule, never from irregularity: one maximally general rule doing the work natural languages spread across special cases and their exceptions.

**Exceptions dissolve because the grammar is data and irregularity pays admission.** The grammar is enumerable — every token, construction, and precedence a queryable node (*Meta-reflection and the unified system*) — so there is no place for an implicit rule to hide. A form as irregular as any natural-language idiom is admissible, but only as an explicit constructor, admitted under the recognizer's proof obligations (terminating, deterministic, effect-free) and disambiguated by a deterministic order, never a statistic (*Identity recognition*). An exception is therefore not a violation of a rule; it is another first-class rule, visible to every speaker and every tool. The exception category does not exist.

**Generativity is two-level.** A natural language's finite grammar generates infinite sentences; Englogos's finite kernel additionally generates grammars. Constructors are Turing-complete, so the surface can grow — new constructions, new vocabulary as new identities — while every extension remains expressed in, and checked by, the same fixed core. Lexicon and grammar are both open the way a natural language's lexicon alone is; the kernel underneath never moves.

**Meaning is the proof layer's, exactly as for hosted text.** A declarative Englogos sentence denotes a proposition; "executing" it is the checker applied to that proposition; empirical claims enter as asserted axioms with provenance, committing to branches of the ecumenical proof system, and epistemic status — proven, refuted, plausible-to-a-degree, unknown — is assigned lazily and revisably by the coherentist layer, never by the parser. What Englogos adds over hosting is only that the sentence-to-proposition mapping is authored and regular, so a sentence's candidate set is small or singular by construction rather than recovered from irregular text.

**Density through underspecification stays legitimate.** Part of natural language's compression comes from ambiguity resolved by context. Englogos keeps the trick without the guessing: an underspecified form parses to the candidate structure it denotes — alternatives held as plain value structure, per the hosting ruling — and resolution stays lazy, at a use site, surfacing `?` where demand outruns knowledge. The recorded rejections stand unchanged here: no driver-level backtracking, and underspecification is represented as data, never searched for a winner the source does not determine.

**A surface option, not an ethos.** Per the walled-garden warning (*The image's failure mode is designed out*), Englogos is one surface over the graph among several — hosted natural languages and ordinary Logos stand beside it — never a requirement for entry. And the substrate's guarantees end where human-language design begins: learnability, speakability, morphology, and vocabulary design are disciplines of their own that the graph does not answer, which is much of why this section records a direction and not a design.

## Chain programmability: constrained targets as backends

*(Design direction reached in discussion, July 2026; far-horizon, dependent on the backend interface beyond Cranelift, the rewriting engine, and the verification strata, and not on the critical path. Prompted by covenant-style programmability arriving on UTXO chains — Kaspa's Toccata upgrade and its SilverScript language — but recorded chain-neutrally.)*

**A chain VM is a gate-constrained backend, not a new subsystem.** Per *Backends are identities*, a backend carries its own lowering rules and declares the fail-closed gate predicate over the scopes it accepts. A chain's script or covenant VM is exactly such a target: tiny, deterministic, fee-priced, rejecting most of the language. What the gate demands is what the chain demands — determinism, termination, no I/O — and that is already native vocabulary: capability tracking and totality obligations (*The logos strata and verification*), stated and checked rather than left as compiler discipline. Hosting an existing chain language's full semantics as a Logic Graph dialect stays what *Native interop and the Rust bridge* already calls it, a separate and heavier option; the in-grain path is a Logos surface lowered through a chain backend.

**Fee schedules are cost functions.** On-chain execution is priced per operation — gas, script pricing — and the rewriting engine already extracts the cheapest proven-equivalent form under a user-supplied cost function (*The rewriting engine*). A chain backend therefore ships its fee schedule as a cost function and gets fee minimization as the ordinary extraction step, safely aggressive because every rewrite carries its equivalence proof, in a domain where a miscompile loses funds.

**The domain's economics reward the verification strata.** Deployed contract code is immutable, adversarially attacked, and its failure is unbounded financial loss — the inversion of the usual cost-benefit that makes proofs a luxury elsewhere. The historical exploit classes — reentrancy, overflow, access control, accounting errors — are mostly statable invariants, refinement- and proof-layer territory, and the pay-as-you-go strata are the difference from today's bolted-on specialist verifiers: verification in the same language, over the same graph, from a refinement up to a dependent proof. Comptime metaprogramming supplies the flexibility the constrained target cannot: full Logos runs at build time to generate the small artifact the gate admits (*Build and run are one self-directing pass*), so expressiveness lives in authorship rather than being asked of the chain.

**The reach is exact.** The chain VM is a hard ceiling on what deployed code can do; the improvement is in authorship — correctness and cost — never in on-chain expressiveness. Proofs establish only what is specified: wrong specifications, oracle and economic attacks, and key theft stay untouched. And a ZK execution proof is a different artifact from a compile-time proof about the program, though a zkVM fits the backend interface like any other target.

## Feasibility and effort

The accounting of relative effort and risk:

- **Bootstrap seed (Rust)** — bounded; the hard part is restraint, keeping it small enough to audit.
- **Base language (self-hosted)** — the bulk of the language work, but well-understood territory (borrow checking, inference, data structures) and strongly accelerated by LLM-assisted implementation.
- **Rewriting engine** — bounded; the algorithm is established (egg-style equality saturation) and adaptable to the Logic Graph. Cost-function infrastructure and rule authoring take longer to get right.
- **Symbolic math stdlib** — the most uncertain. Differentiation, basic integration, polynomial arithmetic, and simplification are achievable early; Mathematica-level coverage is a long mathematical-engineering effort. Logos need not reach that bar to be valuable — an 80%-case library with compile-time symbolic evaluation is more useful than another runtime CAS clone.
- **Refinement logos + SMT** — established design space (Liquid Haskell, F\*, Dafny); integrating SMT discharge with the base logos stratum and building tooling is real but bounded.
- **Full dependent logos + proof terms** — the deepest, longest piece, building on the same foundations Lean matured over years. Not needed for an initial release; the foundation can be architecturally present while the implementation accretes.
- **Backends** — Cranelift and LLVM are well-documented with many existing integrations to learn from; custom backends (embedded, WASM, GPU) are smaller projects given a good interface.
- **Standard library** — ongoing for the language's lifetime; the pre-release subset must cover collections, numerics, I/O, concurrency, basic symbolic math, and verification utilities.
- **LSP server** — less than building from scratch elsewhere because the information is already in the graph.
- **Documentation generator** — modest; the graph carries everything, the work is rendering and cross-references.
- **Structural editor / IDE** — large, but far smaller than equivalent IDEs elsewhere; not on the critical path.
- **Debugger/profiler integration** — standard-protocol integration first; Logos-native versions later.
- **Compiler-extension framework** — partly free from meta-reflection; conventions and tooling mature alongside the standard library, which demonstrates the patterns.
- **Verified demonstration projects** — a verified kernel module, crypto library, or compiler pass; research-grade, and what establishes credibility in the verification niche.

**Accelerants Logos has that earlier language projects lacked:** LLM-assisted implementation, mature compiler infrastructure (Cranelift, LLVM, egg), well-understood foundations (dependent type theory, SMT, borrow checking), and a decade of accumulated PL research. None make the project trivial; each meaningfully reduces effort versus inventing each piece.

**Decelerants:** the unification ambition is genuinely novel in the systems-language space, so some design problems surface through implementation rather than prediction. The Logic-Graph-contains-everything commitment is tested by every feature, and keeping it coherent demands sustained architectural discipline. The verification stratum is where unexpected issues are most likely.

## Closing

The vision is large, and honest about being large. Each piece has working precedents elsewhere — Lean 4's self-hosting, MLIR's layered IR, egg's equality saturation, Rust's borrow checking, Smalltalk's malleable system — and the contribution is unifying them rather than inventing from scratch. The risks are real but the architecture is sound and coherent.
