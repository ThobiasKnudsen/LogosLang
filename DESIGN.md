# Logos: A Unified Programming System

*The consolidated design document for Logos: vision and architecture first, representation and subsystem detail below. It reads top-down by altitude. Some later sections (identity recognition, representation internals) describe target architecture not yet validated against a real corpus; those are flagged inline.*

## The core vision

Logos's foundational commitment is **radical unification**: every piece of logic the system contains — programs, types, proofs, compilation rules, optimization passes, the compiler itself, documentation, tooling, and the language's own parsing rules — lives in a single data structure, the **Logic Graph**. There is no separation between "the language" and "what is written in it." Everything reachable in the system is queryable and manipulable thrclaudeough the same meta-reflection.

The bet is that the usual boundaries — language vs. compiler, code vs. specification, program vs. proof, source vs. tooling — are accidents of how systems were historically built, not necessities. Collapsing them yields a system that is simpler at its core, more expressive in what it can state, and more honest about what it is.

Logos is also a serious systems language: static memory safety via a borrow checker (lexical lifetimes, no GC, explicit ownership and moves), zero-cost abstraction, capability tracking for effects, and a layered architecture compiling to multiple backends. The unification does not cost much performance. Logic Graph code is interpreted by default; an immutable (frozen) region can be explicitly JIT compiled with Cranelift, which is faster in most cases, whereas mutable code stays interpreted — compiling it would only be invalidated the moment its structure changed. Reflection needs an LG form to exist, not a compiled one — it cannot be done on anything lower than the LG — so JITed code stays fully reflectable through the LG it was compiled from, and interpreted code simply *is* its LG. The seed is the one running thing that is neither interpreted nor JITed: it is compiled Rust that behaves like frozen, JITed Logos, never mutated at runtime (it is all compiled, and one would not want to borrow every seed identity). Its reflectability is a target reached incrementally, not a v1 property: v1 ships the seed's primitives as `native` — callable-only machine code, opaque to reflection — and each becomes reflectable as LG only once it is ported to Logos source, so the seed's logic reflects in full only at the self-hosted end state (see *The architecture*).

## The architecture

**The Logic Graph is the primary representation.** Not an AST (syntax-only) and not a conventional IR (post-elaboration codegen form), it is the unified structure holding the program with all semantic information attached (resolved scopes, inferred types, borrow states, propagated capabilities), the rules that governed its parsing and elaboration, the standard library, the compiler's own logic, and eventually proofs about all of it. Navigation is uniform: the same operations users employ on their own code traverse any subgraph, including the compiler's.

**A small Rust bootstrap seed starts the system.** The seed is the minimum: a parser producing Logic Graph nodes, an evaluator for those nodes, enough type machinery to check the kernel, and a path to Cranelift. Everything beyond — the full type system, borrow checker, rewriting engine, optimization passes, standard library — is written in Logos and processed by the seed, until the system compiles itself. The seed's small size is what makes Logos formally tractable: auditable by hand, eventually verifiable, a fixed reference point. Once built it is compiled and immutable: its own logic is reflectable as Logic Graph — tooling can read it — but frozen, never rewritten at runtime, and it is the single trusted core every other part is processed by and traced back to. The discipline is restraint — knowing what belongs in the seed and what does not. The core the seed realizes is not bootstrapped sequentially: its identities — the node cell, `Type`, `Scope`, `struct`, the primitives — are mutually dependent and come into existence together, one hand-built cyclic graph. Sketches of the core are therefore *specifications* of that graph, not programs (their textual order is not evaluation order), and checking the seed's graph against them is a planned verification step.

**The reflection boundary is callable-versus-data, and it bottoms out in `native`.** What reflects and what does not divides along *callables versus data*, not seed versus user. Every Logic Graph structure — types, values, scopes, and a function's own source body — is reflectable, since reading it is exactly what the interpreter and the lowering do; what is opaque is the *callable*, a machine-code body (`exec@`) one invokes but cannot read into. This is forced, not chosen: interpretation consults each node's evaluator, so were every evaluator itself Logic Graph to interpret, the recursion would never bottom out — some functions must be machine code or nothing runs, and compilation exists precisely to produce them. A function is thus two things at once — a reflectable source body and, once compiled or native, an opaque callable — the same duality as interpreted code being its own Logic Graph while JITed code stays reflectable through the source it compiled from. A **`native`** function is the limiting case: callable-only, a machine-code body with no Logos source, the opaque leaf the tower rests on. The seed ships every primitive as `native`, which is what makes the bootstrap tractable — no spec-Logic-Graph is owed for the primitives, so the seed-against-spec verification above is deferred, not cancelled, returning per identity as that identity is ported. Self-hosting then replaces `native` primitives with Logos source one at a time, each gaining a reflectable body while keeping its callable, so the boundary never moves and the `native` set only shrinks toward an irreducible floor — `invoke`, allocation, raw memory, syscalls — that can never have a Logos source.

**The Logos IR is the codegen-oriented intermediate layer.** Between the Logic Graph and any backend sits a target-agnostic IR with explicit basic blocks, control flow, memory operations, and types. Target-agnostic optimizations (constant folding, CSE, inlining, monomorphization) operate here. The Logos IR is itself representable in the Logic Graph — another stratum of the same structure. The Logos IR should be designed to fit many different backends, like Cranelift, LLVM, WASM, and potentially unique hardware. All of these parts are not the same set so there must be a way to describe what is possible and not for each backend. And the Logos IR should be extensible for the future. But the v1 Logos IR should at least be solid for most/all things Cranelift is solid at.

**The backend interface defines what a backend must provide.** Anything that consumes Logos IR and emits code — Cranelift, LLVM, a custom embedded compiler, WebAssembly, a GPU backend — satisfies one interface. Cranelift is the default (fast compilation, good code, ideal for meta-circular execution); LLVM is selectable for production and hot loops; backends mix within a program at function or module granularity via scope modifiers like `backend cranelift (...)` or `optimize_with llvm (...)`.

## The rewriting engine

Logos collapses three things other languages keep apart — compiler optimization, computer algebra, and user-defined transformation — into one operation: take a Logic Graph or Logos IR fragment, apply rewrite rules, extract the form minimizing a cost function.

The engine uses **equality saturation**: rather than rewriting destructively in priority order, it builds an e-graph of forms equivalent to the input under the rule set, then extracts the lowest-cost form. Extraction is optimal over the forms the e-graph actually holds; reaching a true fixpoint is not guaranteed — size-increasing rules can grow the e-graph without bound — so the engine runs under an explicit iteration/size/time budget and extracts the best form found within it. (Optimal extraction under a sharing-aware cost function is itself NP-hard, so the default extractor is heuristic, with an exact mode available.) The e-graph approach is production-validated — Cranelift's mid-end uses one — though production systems bound the search rather than fully saturating, as here.

- **Rule sets are first-class Logos values.** A rule is a pair of Logic Graph patterns (match, rewrite). The standard library ships canonical sets for arithmetic, algebra, trigonometry, and compiler optimizations; users compose, extend, or override them.
- **Cost functions are first-class Logos values.** A cost function maps Logic Graph nodes to costs. The standard library ships FLOP count, numerical stability, code size, register pressure, and target-specific metrics; users supply their own.
- **Multiple targets, one engine.** Minimum-cost (default `simplify`), canonical (equality testing), factored (root extraction), expanded (pattern matching), numerically-stable (Horner, log-sum-exp), and target-specific forms are each just a choice of cost function and/or rule subset.
- **Compile-time evaluation is first-class.** Because `eval` over Logic Graph fragments is built in, the engine can run at compile time on symbolic expressions and bake only the optimal form into the binary. Differentiating an expression, simplifying the derivative, and emitting native code is one `eval`-bracketed pass with no runtime symbolic machinery. The same applies to optimization: `eval (optimize_for(target, kernel))` performs target-specific rewriting on user code at compile time.

The same engine serves the compiler's `x + 0 → x` and the mathematician's `sin²(θ) + cos²(θ) → 1`. The compiler's optimization passes are Logos libraries over this engine, not hand-written compiler code; adding a pass is library work.

## The type system and verification

The strata compose, and users pay only for what they use.

- **Base type system** — primitives, structs, enums, arrays, function types, references with lexical lifetimes, ownership and moves, capability annotations (`pure`, `total`, `async`, `thread_pinned`, GPU-subset, allocator-bound). The borrow checker operates here: for whole values and fixed places (fields, elements) it is decidable, fast, and points errors at nodes the user wrote; borrowing a *predicate-defined* set of indices is the advanced case, which discharges its disjointness obligation through the SMT layer and is therefore bounded by the solver, not constant-time. One rule covers every case: among references that are simultaneously live and overlap, many readers XOR one writer — read∩write and write∩write empty, read∩read free. The checker is therefore one generic function asking one question — *can these two extents overlap; does this access fall inside this extent?* — answered yes, no, or unknown; the tiers are not separate rules but answerers of rising strength for the same question (identity and field-offset comparison; constant then symbolic index arithmetic via SMT; predicate disjointness where decidable; kernel-checked proof beyond), and what varies between borrow-checking "cases" is never the rule but the difficulty of computing its inputs: which references coexist (lifetimes, control flow, threads) and which references exist at all (summaries at function boundaries, since references escape through data structures).
- **Refinement types** — predicates on primitives (`int32 where >= 0`, `int32 where 0 <= self < array_len`), with an SMT solver discharging obligations at assignments and call sites. Rules out out-of-bounds, division-by-zero, and overflow at compile time wherever the solver can discharge the obligation; where it cannot, the obligation surfaces as an error to resolve — a strengthened precondition, an explicit check, or a handled case — never a silent pass. Code using no refinements is unaffected.
- **Pre- and post-conditions** — refinement predicates on function inputs and outputs; preconditions proven at call sites, postconditions in bodies. They compose: a call yields its postcondition as a new fact.
- **Termination annotations** — opt-in decreasing measures for functions that want to participate in proof contexts. Functions without them stay usable but cannot appear where totality is required.
- **Full dependent types and proof terms** — types depending on values, propositions as types, proofs as programs, checked by a small trusted kernel. Arbitrary mathematical theorems about programs become statable and provable.

**Verification composes with rewriting.** Each rewrite rule can carry a proof that it preserves the relevant property (equality, refinement satisfaction, numerical equivalence within tolerance); the engine then guarantees simplification preserves it. A verified CAS — every simplification provably equality-preserving — emerges as a library.

A systems programmer gets the base type system and borrow checker; someone wanting guarantees beyond memory safety opts into refinements and pre/post; scientific, cryptographic, or verified-systems work opts into dependent types. The strata interoperate; parts of a program can be verified while the rest stays lower.

## The proof layer

**Proofs are a layer over the Logic Graph, not a property of the substrate.** The base node model classifies a value by following its `type` pointer — a recursion that terminates in the self-referential `Type : Type`. That self-loop is sound as *operational* machinery (it is never read as a logic) but inconsistent as one (Girard's paradox), so the proof system does not build on it. Instead, propositions, proofs, and their classifying universes are ordinary Logic Graph values that **carry their logical universe as stored data, read by `universe_of` rather than `type_of`**. The same nodes thus bear two classifications — `type_of` for execution (self-looping, irrelevant to the logic) and `universe_of` for checking (stratified, stored) — so the proof system's soundness is decoupled from `Type : Type` entirely. This is the standard kernel treatment of universes (levels as checked data, as in Lean/Coq) recast in the node model.

**Universes are stratified; propositions are erased.** `universe_of` enforces a predicative tower — `Universe 0 : Universe 1 : …`, generated lazily, never `Universe : Universe` — which is what excludes the paradox. `Prop` is the distinguished universe of propositions: impredicative, proof-irrelevant, and erased before codegen, so verification costs nothing at runtime. A proposition is a value in `Prop`, a proof is a value inhabiting it, and the connectives (`False`, `And`, `Or`, implication, `Forall`, `Exists`, equality) are ordinary inductive types and function types — no proposition primitives. Curry–Howard holds at the *representation* level (a proof is a real, reflectable typed node) while logical typing is the separate `universe_of` relation, giving the isolation of a deep embedding without demoting proofs to inert data.

**The trusted obligations are small and explicit.** The kernel verifies each stored universe rather than trusting it (enforcing stratification), checks strict positivity on inductive definitions, and confines definitional equality to the terminating proof-layer fragment — the same tag that distinguishes a proof object from substrate junk gates what the checker may evaluate, so it can never be driven to reduce a non-terminating substrate term. Two walls follow: nothing eliminates from the `Type : Type` substrate into `Prop`, and conversion reduces only tagged, terminating objects.

*(June 2026 review — open decision. If `Prop` stays proof-irrelevant, one obligation is missing above: elimination from `Prop` into a computational universe must be confined to subsingleton propositions (Lean/Coq's large-elimination restriction), or proof irrelevance is unsound. The alternative under consideration is **proof-relevance** — proofs as ordinary, distinguishable, reflectable graph values, with erasure demoted from an automatic property of `Prop` to an ordinary optimization — which needs no such restriction and fits both the reflect-everything stance and the ecumenical layer below. Resolving this fixes the proof-irrelevance wording in this section.)*

**Theorems, axioms, and reach.** A theorem is a named proof — a definition whose type is a proposition and whose value is a kernel-checked term. An axiom is a proposition declared with an empty value slot: asserted, not proved, enlarging the trusted base, with every dependent proof traced back to it. The sound layer can prove properties of *any* Logic Graph node — the self-referential substrate and the proof system itself included — by reasoning over representations and an operational model, treating substrate types as syntax and never lifting substrate-typing into logical truth (the CompCert/MetaCoq discipline). It verifies everything above it; by Gödel's second theorem it cannot prove its own consistency, which stays the irreducible part — trusted by hand-audit, not proved.

**The layer is deferrable.** None of this is needed for the substrate to run, so the proof system ships after Core, in the standard library. Its smallest checking core — the proof kernel — is an ordinary importable library, not part of the seed: it is trusted only by code that opts into it, so it never enlarges the seed's mandatory trusted base, which stays the single core everything is processed by. Everything stated and proved on top — the connectives, theorems, and tactics — is *checked* by that kernel, not trusted.

## The ecumenical proof system

*(Design direction reached June 2026; like the rest of the proof layer it ships after Core and is deferrable.)*

**Proofs do not live in one global theory; they live in a graph of theories.** A single consistent axiom set is one *branch*; the Logic Graph holds many, including mutually contradictory ones — `P` in one branch, `¬P` in another — each internally consistent, coexisting without poisoning each other. A contradiction arises only from combining incompatible assumptions in *one* proof, and the system never does: every proof carries the **set of axioms it depends on** (its branch), and two results combine only when the union of their axiom-sets is consistent. This is the **ecumenical proof system** — named after the proof-theoretic *ecumenical systems* in which classical and intuitionistic logic coexist in one framework — and structurally a *theory graph* (cf. MMT, Hets, the IMPS "little theories" method) that carries its own dependency provenance.

**The branch is the axiom-closure, not the whole dependency-closure.** Definitions are branch-neutral and shared; only *axioms* — propositions asserted with an empty value slot — commit a proof to a branch. So when the kernel checks a proof it emits the set of axioms that proof transitively rests on (as Lean's `#print axioms` does), and that set is the proof's world. Adding a branch *soundly* means forking on an **independent** statement — one the base can neither prove nor refute, like the parallel postulate over Euclid's other axioms, or CH over ZFC: you *assume* `P` or *assume* `¬P`, never prove both from one base (which would mean the base itself is inconsistent).

**Consistency is checked, never guaranteed.** Keeping `P` and `¬P` apart is trivial — one is the other's negation. But two axioms that look independent yet are jointly inconsistent derive `False` only through some chain, and finding that chain is undecidable (a system cannot prove its own consistency — Gödel's second theorem). So a combination is admitted as compatible **until a `False` is derived from it**, whereupon the cluster is marked poisoned and its results withdrawn — non-monotonic, in the manner of assumption-based truth maintenance and belief revision. "Consistent" means "no contradiction found within budget," never "proven impossible."

**A coherentist layer ranks plausibility, and it is read-only.** Backward search reduces a goal to the sub-statements that would establish it, recursively, and *where it bottoms out sets the status*: at axioms or proven theorems the goal is **proven** (certain); at a web of unproven-but-densely-interconnected statements it is **plausible** (a ranking, not a proof). The more consistent structure that depends on a candidate axiom, and the more it interconnects, the higher its plausibility — coherentism computed over the proof graph's own edges. These scores feed *what to try proving*; they **never** count as proofs — the same one-directional wall as the rest of the layer (kernel → ranking, never ranking → kernel). This is the system's humility: it self-corrects the instant a proof or a contradiction arrives, while keeping every still-consistent branch alive.

**It composes with the universe kernel rather than replacing it.** The two are orthogonal layers over the one graph: *universes* stop self-referential paradox **within** a theory (the stratified `universe_of` tower above); *branches* manage incompatible assumptions **across** theories. They meet at one productive point — large-cardinal axioms, which assert that taller universes exist, so there "which branch am I in?" and "how tall is the universe tower?" are the same choice, settled (as in real mathematics) by fruitfulness, i.e. by the coherentist layer.

## Memory and concurrency

- **Stack by default** — locals are scope-bound and dropped automatically. No GC.
- **Explicit heap** — `alloc T v` returns an owning `ptr T`; freed via `drop`, scope-end, or `defer` (LIFO). `own` transfers ownership and removes the source identity. `&T` / `&mut T` are statically checked (many shared OR one exclusive), with lexical lifetimes keeping the model predictable.
- **Place-granular borrows** — borrows are tracked per *place* (a field, an element, a predicate-defined set of indices), not per whole object, so the borrow rule (many shared, one exclusive) applies only where two places provably overlap — a disjointness obligation the checker discharges through the refinement/SMT layer, being itself a Logos program free to query the solver. Exclusive and shared borrows of disjoint regions of one object therefore coexist — odd indices writable while even indices stay shared-readable, say — capturing safely the split-borrow patterns conventional checkers reach only through `unsafe`. These borrows are *local*: resolved at the borrow site, within the scope that can see the place, and they do not outlive it carrying their extent — the escaping case, where the extent rides on the reference identity itself, is *Reference-granular permissions* below.
- **Reference-granular permissions (post-v1; v1 must not block them).** The fine-grained extent of a borrow is carried by the *reference identity itself* — a borrow is already its own node — as a total, pure predicate over places. The obligations it generates (disjointness between coexisting references, containment of each access) discharge statically through the tiered ladder above, and predicates and proofs erase before codegen like everything in `Prop`, so a fully discharged reference compiles to a bare address; a runtime check is only ever an explicit opt-in fallback, never silently inserted. What v1 owes this future: references exist as graph identities at check time and erase to raw addresses in codegen; the extent descriptor is deviation storage (the default whole-place reference stores nothing); the checker is factored around the one overlap/containment query so later tiers plug into its *unknown* bucket; places are graph values a predicate can mention; the optimizer keys aliasing and immutability on a reference's extent, never implicitly on whole objects; and extent predicates are admitted under the same obligations as recognizers — total, pure, terminating.
- **Read and write are one mechanism across the system.** Visibility (`pub`), mutability and borrowing (`&` / `&mut`), and graph access (reflection and `graph_mut`) are not separate features but one primitive — a read or write *capability over a place, for a lifetime, granted to a scope*. Visibility is the `'static`-lifetime read of a name; a borrow is the region-lifetime read or write of a value or sub-place; reflection and graph mutation are the read and write of a graph node. The borrow rule — many shared XOR one exclusive — is the reader–writer rule, and it governs all of them uniformly, including across threads. The scope tree supplies the default grants (a declaration is reachable in its own scope and descendants), so the common case needs no annotation and `pub`/`mut` remain sugar over the primitive. One model, two enforcement timings: static checking for *accessing* fixed structure, incremental re-validation for *mutating* it — because writing the graph changes the structure the checker reads. The checker holds a privileged read over the whole graph — it must see every node to identify and check what is built on it — so user-facing visibility restricts *peer* access, never the checker's reflection (a private field is hidden from other code, not from the checker). Beyond the coarse per-branch grants the scope tree gives, access may also be set by *functions* — total, pure predicates over places or scopes — for the fine-grained cases (these indices writable, that field readable only by its type's methods). These access-setting predicates are **`gate`s**, ordinary identities like any other — `pub` and `mut` are themselves gates. A gate is a **fail-closed predicate** returning `true | false | ?` (the universal unknown): access is permitted only on `true`; `false` and `?` both deny, with `?` retained as a standing obligation — the synthesis goal a later prover discharges, moving an access from `?` to `true` by proving it safe (composition is Kleene three-valued, so an unresolved `?` stays visible, and only the enforcement boundary collapses `?`→deny). The check is **lexical and static** — decided from the viewer's scope chain at elaboration time and erased before codegen — so there is **no ambient authority**: being *called by* a holder grants nothing, an indirect call cannot launder access, and the only way to give a callee access is to hand it the reference explicitly. A gate's requirement propagates *backward* through the call graph as a static obligation, discharged at the scope that legitimately grants it, and it stays analyzable starting from any code section. The seed builds the chokepoint, the lexical tier, and the typed `?` seam only — **no prover**; the stronger answerers (the borrow checker, the proof layer) bolt onto `?` exactly as the tiered checker above describes.
- **User-replaceable allocators** — `alloc.arena`, `alloc.bump`, `alloc.pool`, `alloc.system`, and custom allocators share one interface. Arenas enable cyclic structures via indices rather than borrows.
- **Two concurrency shapes** — `parallel for` for data-parallel work over disjoint indices (the borrow checker recognizes the pattern); async for I/O-bound concurrency via stackless tasks on user-defined executor pools. A function that can pause is marked `async` and pauses at an explicit `.await`, so suspension is always visible in the source, and `.await` is allowed only inside an `async` function; the `async pool (...)` modifier selects which executor pool runs a region. Tasks are anchored once started (no Pin machinery); borrows cannot cross awaits except within the same task's anchored state struct; capabilities flow through the call graph to enforce this statically.
- **Scheduling, preemption, and cancellation.** A task yields cooperatively at each `.await`; beyond that, the runtime can step in at any *boundary* where the task's live state is materialized as graph data — between interpreter steps, or at the edge of a compiled burst. This is what keeps one task from freezing its pool: instead of one opaque compiled function holding state in registers across the whole task, a hot region may compile to short bursts that read their inputs from the task's live state, run at full speed, and write results back at each boundary, so that state is ordinary graph data at every boundary. Preemption and cancellation are then uniform and cheap — a paused or preempted task *is* its materialized state, with no stack maps to emit and nothing to reconstruct from registers; cancelling is dropping that arena-scoped state like any other runtime graph, running the ordinary drops on its live locals. Only the transient within-a-burst work is ever held in registers, and it is never seen from outside because control is taken only at boundaries. Boundary granularity is a per-region dial: short bursts give responsive preemption and cancellation, at the cost of writing live state out each boundary rather than keeping it in registers and of optimizing only within a burst; one large burst — a whole hot loop — gives top throughput but few stopping points, reserved for compute kernels, the longest burst bounding the worst-case time-to-preempt. It rides on the same interpret/compile boundary the representation section describes (a compiled region falling back to interpretation): where an artifact can deoptimize is where the scheduler regains control. Preempting is invisible and the task resumes seamlessly; cancelling stops it at its last boundary for good — its values are dropped but its remaining steps never run, so keeping state consistent across a possible cancellation is the program's responsibility. *(Scheduling model reached in design discussion, not yet validated; the bootstrap can ship cooperative `.await` yielding and add boundary preemption as the compiled tier matures.)*

## Mutability and construction

*(Mechanics settled in design iteration, June 2026; declaration syntax and the `mut` model revised later that month — `mut` recast as a type modifier (mut on the type denotes a mutable value), not a positional slot qualifier — applied recursively up the classifier tower, with value- and type-mutability as independent permissions on the two slots. The phase-bit storage is proposed, not yet validated, and whether everyday value-mutation rebinds to a fresh node or writes in place is still open.)*

**Declarations are immutable by default; the grammar mirrors the cell's two slots.** A node *is* a type slot and a value slot, and three operators write them: `:` is **declaration** — it introduces a name and sets its type slot (`key : type`), leaving the value `undefined`; `:=` is **declaration with a value** — it introduces a name and sets its value slot, inferring the type from the value (`key := value`); and `=` is **reassignment** — it writes the value slot of an *already-declared* name (`key = value`). The colon writes the type slot, the other two write the value slot; they compose as `key : type = value` (the `:` fixes the type, the `=` then fills the value). A bare `key :` declares a name whose two slots are both `undefined`, and an anonymous typed value is written by juxtaposition — `i32 32`, `f64 2.0`, the type preceding the value — never `type = value`. The relation is exact: a name may carry a type and a value but need not, a type may carry a name and a value but need not, and **a value always carries a type** (the type defines how it is read). Names live in the trie; type and value live in the node.

`=` is **reassignment, not declaration**: `key = value` overwrites the value of an *already-declared* `key`, keeping its type, and is an error if `key` was never declared — so a mistyped `cont = 6` on an undeclared `cont` is caught rather than silently creating a binding. Declaration is therefore always `:` or `:=` (`key :`, `key : type`, `key := value`, `key : type = value`); `=` only ever rebinds what already exists, and type inference rides `:=` (the type is read from the value).

**`mut` is a type modifier — `mut T` is the type of a mutable T**, so mutability is something the *type* states about the value, not a separate value-position qualifier. The type is the sole authority on what the value permits, so the capability lives there: a value typed `mut T` is the everyday mutable variable, one typed `T` is immutable, and there is no value-position `mut` form — `mut` never sits before the value (`i32 mut 32` is rejected: the value is raw bytes, with nowhere to carry the capability). This is the very same `mut` as in `&mut T` (mutable access to a T), one rule rather than two, which also retires the old inconsistency of writing `mut` on a reference type but forbidding it on a declared type. A struct's per-field mutability rides on each field's type the same way (`f : mut F` mutable, `f : F` not), so the type alone describes which parts are writable. Changing the type slot itself — retyping the node rather than writing its value — is not a value-level mutation: it needs the type's *own* classifier to carry `mut` (the identical rule one level up), or a fresh `dyad` built with the new type, which is the immutable-by-default path. This is one recursive rule: **`mut` left-prefixes a type at any level of the classifier tower (`value : i32 : type : …`) and makes the level just below it mutable** — `mut i32` marks the *value* mutable, `mut type` (mut on the type's own classifier) marks the *type slot* mutable. The two are independent permissions on the two slots, so a value-construction expression spells out four states: `i32 32` (frozen type, frozen value — the plain constant, by far the most common), `mut i32 32` (frozen type, mutable value — the ordinary variable), `mut type i32 32` (mutable type, frozen value — *reinterpret-only*: the fixed bytes may be re-read through another type but never overwritten), and `mut type mut i32 32` (both — full structural mutability). Making the type mutable does not imply a mutable value; reinterpret-only is its own quadrant. The tower tops out at the `Type : Type` self-loop, so `mut` sits at only these two levels in practice — anything higher would mutate the universe. Prefixing the *name* with `mut` (`mut key`) still makes the **name/identification** mutable — the trie entry, not the node, since the node's content is already reachable through its two slots — and `mut (…code…)` still makes a whole region **structurally** mutable. These stratify by how much structure can change, which is exactly the compile/freeze rule: value-mutation is ordinary data change and stays **compilable**; a type change rewrites how the value is read, so it is **checked and deoptimizes** on write; a structural `mut (…)` region has a live structural writer and therefore **stays interpreted** until it freezes.

**Both slots of a node follow one lifecycle: `undefined` → defined → frozen.** `undefined` is the hole — no value yet, writable until defined, and reading it is a checked error, never undefined behavior. The exclusivity rule makes that check local: while a slot is still mutable, only the constructing scope can reach it, so no other scope can ever observe an undefined slot — the flip *is* publication. The two slots obey **one rule at successive levels**: each is written by its definition and then frozen by default, and stays writable past definition only when its classifier carries `mut` — the value while its type is `mut T`, the type slot itself only while the type's *own* classifier is `mut` (the same modifier one level up). The giant common case carries no `mut`: defined once, frozen, the type reading true forever. Retyping a live node is therefore not an everyday declaration — it needs that one-level-up `mut` or a fresh node — and it is not free: a type write must discharge **validity** (the existing bytes are a valid value of the new type) and **layout** (the new type reads the same shape, or storage is reallocated), obligations the refinement/proof layer discharges where it can and that fall to `unsafe` where it cannot, exactly like a checked `transmute`. Because reallocation is itself a value write, a *layout-changing* retype additionally requires the value to be mutable; the mutable-type/frozen-value state therefore admits only layout- and validity-preserving retypes — the bytes are re-read in place, never moved or rewritten. Structural graph edits that rewrite a node's type are the other sanctioned type writes, triggering re-validation as all `graph_mut` writes do.

**The flip is `immut x`, and it is one-way.** Flipping is a write to the node's phase, so the ordinary reader–writer rule blocks it while any reference is outstanding — making it owner-only with no live borrows, the same shape as a move — and no separate authority mechanism is needed (a `metamut` grant was considered and rejected; delegating flip authority is just ownership transfer). The reverse flip is rejected outright: facts derived from reading a frozen node outlive any reference to it — a checker concludes and drops its reference — so un-freezing is unsound without universal dependency tracking. Frozen is final, which is also why a flip never invalidates anyone: no reader can have assumed less mutability than existed.

**Phase is stored as tag bits, not wrapper nodes** *(proposed)*. A wrapper around the node breaks type transparency — reading the node's type would yield the wrapper — and a wrapper spliced into the value chain falsifies the prime invariant that the type defines how the value is read. So the construction phase lives in neither: nodes are sixteen-byte aligned, the low bits of the type pointer are therefore free (four of them), and they record each slot's construction state — undefined, defined, or frozen — for both the type and the value slot; name-mutability and structural `mut (…)` are not per-node phase, living instead in the trie entry and the region marker respectively. The flip clears a bit in the node's own cell — no wrapper ever exists, no address anyone holds ever changes — and the model stays uniform while the storage stays free: every node *has* a phase, none *stores* one as structure. Interpretation masks the bits; compiled regions are frozen by construction, so masking is elided exactly where speed matters. Whether the phase should also be *visible* as structure is a presentation question, not a storage one: reflection can synthesize a wrapper view of a node whose mut bit is set, so tooling that wants mutability in-your-face gets it without every node under construction paying sixteen bytes for the privilege. Per-field mutability of instance fields is carried by neither bits nor wrappers — fields in a value area are raw bytes at offsets, not nodes, under any scheme — but by each field's declared type in the type's scope (`mut F` vs `F`), per the modifier above.

**Two muts, and the storage partition.** The `mut` in a *declaration* is a per-field fact across all instances: it is what the derived layout reads, it is fixed when the type is defined, and an instance's later flip never changes a shared layout. A field that is ever mutable — even only during `init` — is per-instance; a binding that is never mutable is identical across every instance and is stored once with the type. `struct` is thereby simultaneously C++'s struct (its mut declarations, from which the per-instance layout is *derived* rather than authored) and C++'s namespace (its immutable members, stored once). Placement is decidable at the declaration site, writes being the only thing that can differentiate instances. A type's layout-relevant slots must be defined and frozen before its first instantiation; the type locks there.

**Resolution is one rule.** A type keeps the defining scope it was constructed from, and `a.x` asks `a`'s type for `x`; the declaration found decides everything — declared mut: a *place*, the byte offset inside `a`'s value area; immutable: the stored node itself. Visibility is one extra test at the same lookup. A call frame is an instance of its function — parameter declarations resolve to a frame exactly as field declarations resolve to an instance — and lookup continues up the type chain, terminating at the `Type : Type` self-loop.

**Substrate vocabulary.** `any` was removed and split: `undefined` is the hole, carrying a definedness obligation; `?` is the **unknown** — a deliberate, typed hole the author writes in place of code not yet designed, inert until filled and rendering any build that reaches it incomplete by construction; it differs from `undefined` (which says *nothing was written here yet* — a checked error to read) by saying *the value is not yet known*, a standing synthesis goal a maturing Logos discharges by the same backward proof search and coherentist plausibility ranking the proof layer already provides, graduating a `?` from left-alone to suggested to filled-and-verified. `void@` is a type-erased address whose interpretation comes from elsewhere — the honest type of the cell's value field, since interpretation always comes from the type pointer. `native` is a function that is callable-only — a machine-code body (`exec@`) with no Logos source, opaque to reflection (invoked, never read into), and distinct from both `undefined` and `?` because it runs and reaching it is no error; the seed's primitives are all `native`, ported to reflectable Logos source one identity at a time (see *The architecture*). Recorded as rejected, to stay rejected: `metamut`, the reverse flip, and `any`.

## Native interop and the Rust bridge

**Logos reaches existing native code as black-box C-ABI calls, not by hosting other languages.** The runtime is an ordinary Rust program built by rustc (LLVM); Cranelift's role is to JIT-compile Logos code at run time. A foreign function is called in-process, by address, across the platform's native calling convention — the same convention `extern "C"` selects, and the one Cranelift-based JITs like wasmtime reach (through trampolines) to call host functions — so even JIT-compiled Logos code calls native code with no linker step, independent of which backend compiled either side. The boundary is the conventional one — `extern "C"` signatures, `#[repr(C)]` data, panics caught before they cross, an `unsafe` contract the borrow checker cannot see past — and richer Rust types (`Vec`, `String`, generics) are reached through a thin per-function shim a binding macro can generate.

**The toolchain is a build dependency, not a runtime one.** Logos ships as a standalone interpreter; that it is written in Rust is how it was built, not what it needs to run. Native code is compiled in as core builtins, loaded at runtime as separate `cdylib` artifacts (interpreter and Rust toolchain ship apart), optionally produced on demand by invoking `rustc` and then loaded, or isolated behind IPC where a crash must not propagate. Hosting a language's full semantics as a Logic Graph dialect remains a separate, heavier option; the bridge is the cheap path to reusing existing ecosystems.

## Error handling

Errors are values: tagged unions `(T | Error)`. Fallible functions declare their error type; callers handle results explicitly via `match`, with `success` / `fails` combinators for boolean outcomes. No exceptions; no implicit propagation in the core, though a propagation sigil may come as sugar.

## Meta-reflection and the unified system

This commitment ties everything together.

- **The Logic Graph contains its own parsing rules.** Every token, operator, precedence, and associativity is a queryable node; the parser is non-context-free because it consults the trie, which is itself part of the graph. Tooling becomes language-aware by introspecting the graph rather than reimplementing the syntax.
- **The Logic Graph contains the compiler.** Above the bootstrap seed, the borrow checker, type checker, rewriting engine, optimization passes, and the lowerings (Logic Graph → Logos IR → backend) are all Logos programs and all graph subtrees; the seed itself is the frozen exception — reflectable but not rewritten. Adding an optimization is library work; targeting a platform is implementing the backend interface plus contributing rules.
- **The Logic Graph contains the standard library.** Standard-library code is not special; it is code that ships with the system, readable and (carefully) overridable.
- **The Logic Graph contains documentation, source positions, comments, and metadata.** Docs are structured prose interleaved with code; cross-references are graph edges; positions and comments are node properties.
- **The Logic Graph contains proofs and specifications.** A refinement is a subtree of a signature; a correctness proof is a subtree of a definition. Refactoring code and refactoring its proof use the same tools.
- **The Logic Graph eventually contains the IDE.** Editor behavior, highlighting, refactoring, debugging UI, and configuration are Logos programs over Logic Graphs — hot-reloadable, customizable in Logos, with structural editing as library functions. The Smalltalk vision applied to a modern systems language.
- **Stratification handles foundational paradoxes invisibly.** When the type system reasons about itself or proofs quantify over the proof system, a universe hierarchy (following Lean 4) keeps things consistent. Ordinary code never encounters it; only explicit meta-mathematics does.

## Representation, elaboration, and lifetimes

This section records the mechanics of how the Logic Graph is represented, how source becomes runnable, and how runtime-generated structure relates to source.

**A node is a type and a value; its definition supplies everything else.** Every node is exactly two pointers — a `type` and a `value`, sixteen bytes — and a node's *identity is its address*, so nodes are uniformly addressable by handle whatever allocator holds them (no arena is required; the address is the id). The `type` pointer defines how the `value` is read: the type's *definition* — itself a Logic Graph — specifies the node's layout, how it consumes surrounding tokens during parsing, the conditions under which it lowers to Logos IR, and the IR it produces for each well-typed case (a `+` over mismatched or sizeless types simply does not lower until that is resolved). Operands are *not* stored inline; they are reached through the `value` pointer, so a binary `+` is a single 16-byte cell referencing its two operands rather than three inline pointers. What `value` points at is an operand *struct* whose layout the node's definition supplies — the same rule that a type defines how its value is read, applied to operators: a binary operator's value is a two-field record (LHS, RHS), a function call's is its argument struct (one field per argument, a one-field anonymous struct for a single argument), and a variadic call's is that struct sized per call site. The operator is thus a higher-level identity describing *how to read its operands*; which concrete machine operation runs — which `+` — is resolved from the operator together with the operand types, not from the `+` node alone. A node is therefore directly addressable once its handle is held, but the graph is *not* addressable by position or name without context: source code is meaningless without its surrounding context, so lookup from text proceeds by trie-directed traversal (the same trie the non-context-free parser consults), with auxiliary indices added where repeated reflection warrants them. Every node *is* an identity; most identities carry no string name and are addressed structurally, by pattern-matching over the trie and graph, rather than by name. This is MLIR's idea — a node whose *definition* describes its parsing, operands, and lowering — recast onto a uniform two-pointer cell and fused with the reflective parser. MLIR itself is a candidate *backend* behind the backend interface, never the IR: the Logos IR must be a Logic Graph stratum so the same rewriting engine, type system, and proofs apply to it, which an opaque external IR could not provide.

**Numeric literals are uncommitted until context types them.** A numeric literal lexes as `rational_number` — not a fixed machine type but a comptime-number that can mold into any `int`, `uint`, or `float`. Because the concrete `+` is chosen from the operand types (above), a literal beside a typed value takes that value's type — `a + 1` with `a : i32` is i32 addition, `1` molded to i32 — which is what lets one operator and one literal form serve every numeric type without per-literal suffixes (`1i32`, `1.0f`). When *both* operands are still `rational_number`, there is no machine type to mold to, so the operation is carried in arbitrary precision over the number's textual form and the result *stays* `rational_number`, committing to a concrete type only when it finally lands in a typed slot. The seed builds only the mold-to-concrete path (all the `a = a + 1` oracle needs); the arbitrary-precision both-uncommitted arithmetic is deferred, not blocked.

**The cell is self-describing down to one fixed point.** The cell's `type` field is itself typed as a pointer to a *node* — not to a `Type`: that something functions as a type is a checked invariant, not a substrate field type, because the substrate cannot state it. Interpretation is the recursion — to read a node, consult its type; to read that, consult *its* type — grounding out at the one node whose type is itself, whose layout is the seed's only a-priori knowledge. With the value field a type-erased address (`void@`) and the type field a node address (`dyad@`), the substrate is described purely in terms of itself plus raw addresses.

**Source becomes runnable by interpretation; compilation is opt-in.** Source text is converted to Logic Graph nodes one token at a time, and the graph is interpreted directly — there is no separate parse-then-run step once it is in the graph. Scopes marked for a backend (`backend cranelift (...)`) are lowered before they run; everything else is interpreted, and the runtime may promote hot regions to compiled form. Compile-time evaluation is ordinary interpretation performed earlier: a `comptime` marker forces it where the guarantee matters (a result baked into the artifact rather than recomputed each run), but the common case is inferred — the compiler interprets the graph fragments a compiled scope depends on. Because compile-time evaluation runs under a capability set that excludes I/O (effects are capabilities; see the type system), builds are reproducible by construction, while ordinary runtime code retains I/O. Parsing rules are Turing-complete: a type's `token_eval` is unrestricted code — surface forms as irregular as natural language are in scope — and a declarative consumption signature, where a type chooses to carry one, is an optional fast path the recognizer can merge and index, never a ceiling. The generic prologue every `token_eval` needs (bounds, this-token-is-me) is not written per type: `Type`'s own `token_eval` installs it, by reference to one shared checker, into every type it constructs, so the contract is visible graph structure; core identities, hand-built by the seed rather than constructed through `Type`, are correct by construction. Evaluation consumes a *disposable working copy* of the tokens — a plain copy in v1, structure-shared snapshots when file sizes warrant — so `token_eval`'s destructive splicing never touches the persistent token record that incremental identification depends on.

**Compilation is a reader of frozen structure.** A compiled artifact is a long-lived *reader* of a subgraph's structure; what licenses it is the absence of any structural writer over that region for the artifact's lifetime. So the rule that a `graph_mut` invalidates outstanding readers *is* deoptimization — a structural write to compiled code drops the artifact and falls back to interpretation — while ordinary runtime values flow through unaffected, because what froze is the code's structure, not its data. The interpret-versus-compile choice is then an adaptive matter of profitability, not semantics, and needs no syntax: unannotated code is interpreted while cold or unstable and compiled once it is hot and frozen, with `comptime` the one execution marker that is genuinely semantic (it fixes *when* evaluation happens and what is baked into the artifact). The seed, however, does not promote automatically: it interprets everything and compiles only regions explicitly marked for a backend (`backend cranelift (...)`); profile-driven automatic promotion is later work that rides on the same deopt boundary. The same freeze licenses optimization — equality saturation needs a fixed input — so compiling a region, optimizing it, and extracting a cheaper proven-equivalent variant are one operation under one condition: no live structural writer. Editing itself runs only this analysis machinery, which is pure (it holds no I/O capability) over the changing graph; the user's program runs nothing until invoked, and the language's own machinery, being hot and structurally stable, sits permanently compiled and untouched by edits to user code.

**Mutable code is compilable; the reader–writer rule, not mutability, gates compilation.** A region typed `mut` is not barred from compilation — only a region with a *continuously live* structural writer is, and then only because the next write would deoptimize it, so compiling it would not pay. A mutable region that is currently stable compiles and runs at native speed like any other; the "`mut` stays interpreted" rule is this profitability fact, not a prohibition. What governs compilation is the one borrow rule applied to code as data: a compiled artifact is a *read reference* to code structure, so a structural write needs exclusive access and therefore drops every such reader first — that drop *is* the deopt. Two readers of code must not be conflated, because different timings keep them fresh: the **name index** (the trie / resolution structure) is corrected *as part of* the structural edit, so resolution is never left stale — a global shared index cannot be allowed to be lazily wrong; **`bcode`**, by contrast, `run` trusts and does not re-verify per call, so its freshness is the deopt layer's job (null the stale `bcode`, fall back to interpretation), not the writer's. Until the borrow checker makes "no structural write while a code reader is live" static, this exclusion is a discipline the program keeps, exactly as with every other mutation.
**Execution is function application, and only functions execute.** Everything runnable is a `fn`; operators (`+`, `=`), field access, and even control flow are functions, so evaluation has one rule: to evaluate a node, read its `type` — if the type is a `fn`, invoke it on the node's value (its operand struct); otherwise the node is data, read through its type's layout. A node whose type is a function is therefore an application, exactly as the operand-struct model already implies. The `fn` type carries three shared functions — `interpret` (walk the body directly), `compile` (lower the body to machine code, its `bcode`), and `run` (invoke `bcode`) — plus per-instance `input`, `output`, `body` (the reflectable Logic Graph), and `bcode` (the opaque compiled code, null until compiled). `run` trusts `bcode`: it fails if `bcode` is null and otherwise jumps to it, leaving *whether `bcode` still matches `body`* to the checker and deopt layer rather than paying to verify it on every call. The three are simple, predictable primitives; the policy that chooses among them is not theirs.

**Operands travel on the stack, which is why `run`/`compile`/`interpret` take no arguments.** Each reads its own fields (body, signature, code); the call's operands are placed on the stack by the caller and read by the callee's code — the ordinary calling convention. Under interpretation this copies each operand from its node onto the stack per execution (the interpreter's standing cost, as in any bytecode VM); under compilation the node is read once at compile time and operand access is baked into the machine code — immediates inlined, locals assigned to registers or stack slots — so compiled code touches no node at run time and runs at native speed. Because operands live on the per-thread stack and never in the shared graph, execution is concurrency-safe by construction: the shared Logic Graph is immutable code, the mutable per-call state is each thread's own stack frame. This is also what makes recursion work — each activation gets its own frame — independently of, and prior to, any borrow checker.

**A type's metadata is shared by its values; its per-value data is not.** A type stores its `precedence`, `associativity`, `constructor`, and `destructor` once and shares them across every value of that type (its `shared` members); the bytes that differ per value are the per-instance fields. `struct` is the common case of a type whose `constructor` derives the layout automatically — reading the field declarations in its scope and filling `names`, `fields`, and `size_bytes` — so defining ordinary data needs no hand-written layout, while the bare `type` is used where parsing or semantics must be authored (operators, primitives). The two are nearly identical by design, differing only in that a struct auto-derives what a type leaves to its author.

**Build and run are one self-directing pass.** Lexing, parsing, and execution interleave: a construct can `compile` and then `run` a function during the same pass, immediately, rather than deferring it. There is no separate promotion scheduler in the seed — execution is directed explicitly where it is wanted, and adaptive hot/cold promotion is a later layer over the same boundary. A `run` over a scope that compiles and executes on the spot is therefore comptime metaprogramming in full generality — the whole language available at parse time, operating on real Logic Graph rather than a macro sublanguage — and it stays reproducible because comptime runs without the I/O capability, the same boundary that separates compile-time from run-time effects.

**Two graphs, one source of truth.** The *source graph* is file-backed, persistent, and the program's identity; a build is a pure function of it. The *runtime graph* is generated during execution, ephemeral, and owned exactly like other runtime data — arena-allocated (arenas already carry the cyclic structures the graph needs), with the same lifetimes and no garbage collector. Runtime-generated structure is reflected upon the same way source is: by structural traversal, discovered by reachability from the use sites that reference it; there is no value-level or time-travel reflection (a value to inspect is stored in a variable, as in any language; reconstructing past states is the debugger's role over DAP). Reproducibility needs no separate leveling: a build reads only the files, and compile-time evaluation runs without the I/O capability, so the artifact is a pure function of the source. Cleanup needs none either: derived and runtime structure share the lifetimes of the scopes that create them, so when a scope ends its generated graph is freed like any other data, and when the program ends everything generated is gone. The runtime may modify the source, but only on explicit command: the change is a *promotion* — its dependencies are closed over, it must be free of runtime-only inputs (or have them reified as constants), and it is copied out of its arena, unparsed to a source diff, re-checked, and committed under gate. Source is never silently rewritten, and converting any subgraph back to source text (unparsing) is a presentation and serialization facility, not the reflection mechanism.

**Versioned scopes.** A scope may have more than one implementation: the canonical one written in the source, plus any number of derived alternatives — an optimized rewrite, a specialization for known arguments, a different backend — that are proven equivalent and chosen by cost. These alternatives are the structural form of the rewriting engine's equality classes: registered at the scope's entry, extracted by a cost function. No global level numbering or edge classification governs them, because none is needed. A running program may attach optimized variants to a function's live representation and discard them when the enclosing scope ends, exactly as it discards any other runtime data; the source file is untouched unless a variant is explicitly promoted. What gets cleaned up is bounded by lifetimes; what is reproducible is bounded by the files being canonical. (This is unrelated to the type system's *universe levels* — the Lean-style hierarchy that prevents self-referential paradox when proofs quantify over proofs; the shared word names different mechanisms.)

**Concurrent access.** Reading shared graph structure across threads uses ordinary shared borrows (`&`): many threads may read the standard library, definitions, and any subgraph at once. Writes are either disjoint — the `parallel for` pattern the borrow checker already recognizes — or exclusive (`&mut`); concurrent mutation of the same node is rejected at compile time, as elsewhere. Per-node reference counts are deliberately avoided: they leak on the graph's cycles and track liveness rather than the access exclusion threads need; arena ownership governs lifetime instead. The one genuinely shared, mutable structure is the identity interner — allocating or looking up an identity from multiple threads must be synchronized — together with any equality graph that parallel rewriting builds.

## Identity recognition

How the graph answers "what is this?" — for parsing, for reflection, and for rewrite matching — with a single engine. *(Target architecture; the bootstrap ships the trivial version, detailed at the end of the section.)*

**One engine serves parsing, reflection, and rewriting.** Pointed at a node, the recognizer reads the surrounding structure and returns the identities that match there — the operator a token sequence denotes, the construct a subtree forms, the rule whose left-hand side fits. This is tree/term pattern matching — the structural analogue of regular expressions over text, extended with e-matching where the match is over equality classes rather than syntax — and structural grouping, reflective query, and rewrite-rule matching are clients of the one mechanism. (Recognizing tokens from source *text* is a separate, string-keyed layer — a trie over names — not this system; the two are different ways of reaching the same identity, by name and by structure.) Matching is boolean and exhaustive — a node either has an identity or it does not, and the default query returns *every* identity that matches; there is no fuzzy or probabilistic matching in the core. Where a single answer is required (which parse wins), a *deterministic* precedence/cost order selects it — an order, never a statistic.

**Recognizers compile into one shared, stratified structure.** All recognizers merge into one graph; run over a node, it yields the set of every identity that matches there, computing shared sub-tests once rather than re-checking each recognizer separately. The structure is stratified by tractability: a fast *observable skeleton* — navigations and discrete tests on identity, type, arity, and existence — is merged, minimized, and indexed; structural repetition and nesting become cycles and a stack; and arbitrary computation is confined to *guards* at the frontier, run only on the few candidates the skeleton has already narrowed. The skeleton carries the speed, the guards the expressivity. The optimized paths are straight-line — forward dispatch and fail-fast, no backtracking; a condition that compares two nodes is a guard, not a dispatch key, and a relational or negative condition (two operands differ, say) cannot prune the skeleton at all — it rides along as a residual filter or is served by an auxiliary index, exactly as a non-sargable predicate is in a database.

**The structure is itself Logic Graph code — compiled yet extensible without recompilation.** The merged recognizer is not a trie or automaton walked by a special machine; it is ordinary Logic Graph code, run over the start node like anything else, so the same adaptive machinery interprets it cold and compiles it hot. Each identity's recognition code is first minimized and canonicalized — by the ordinary rewriting engine under a compute-cost function, since recognizers are themselves Logic Graph code — so that recognizers branch the same direction wherever they can; equivalent ones then converge onto the same shared paths, and all of them merge into the single graph. Adding an identity is a local merge that reuses those shared paths, hash-consed like any other node, not a global rebuild; the merged graph is allowed to drift from minimal and is re-minimized as a periodic recompaction rather than on every edit. The convergence is guaranteed on the decidable skeleton and best-effort above it, where program equivalence is undecidable and only identical guards merge.

**Admission is a proof obligation, and identification is itself proof.** A candidate recognizer is admitted at a recorded *level* — proven safe (terminating, deterministic, effect-free) or, during bootstrap, not-yet-refuted — and a use-site demands the level it needs, so safety is paid for where required and relaxed where bootstrapping. What is refuted, or resists checking where a use-site demands proof, is rejected; there is no level below not-yet-refuted and no escape past a use-site's demand. This is the proof layer wearing another hat: "X is identity I" is the proposition "X has I's defining properties," so a trivial pattern is a trivial proof and a rich property — totality, well-typedness, a borrow that never conflicts — is a real one; identification, type-checking, borrow-checking, and termination-checking are one activity at different strengths. The system runs that activity on ordinary code, on candidate extensions to itself, and on itself, bottoming out — as everything does — in the small hand-trusted seed it cannot prove.

**Queries specialize the structure.** Beyond the default all-match, a directed query ("is this matrix-addition?") prunes by intersecting the question with the saved recognizers — an automaton product, or in its general form a partial evaluation that bakes in the query's constants — yielding a specialized recognizer that explores only the relevant paths; computing that intersection is itself a job for the identifier. The first pass over freshly tokenized input is discovery (all-match, building structure); directed queries are interrogation over structure that already exists.

**Identification is incremental.** The token array is persistent: an edit changes only the tokens it touches, and re-identification follows the reader–writer rule — each cached identity is a reader of the tokens and rules it consulted, and a write invalidates exactly its readers. Most edits are local; the two that are not are handled explicitly. Redefining an *identity* ripples to its uses, found without a full scan through per-subtree *use-sets* — a summary of which identities occur below each node, carried as one more component of the bottom-up state. A structural edit (an unclosed delimiter) is handled instead by incremental re-lexing forward to the point where the token stream re-synchronizes. (Evaluation's destructive `token_eval` splicing never touches this persistent record — it operates on the disposable working copy described in the elaboration section.)

**This is the target architecture, not the seed.** The bootstrap ships the trivial version — recognizers run directly and unmerged, guards interpreted — and grows toward the fully merged, minimized graph, e-matching, and specialization as measurement on real graphs warrants, never before. The shape it grows into recurs throughout the system: a fast, decidable, canonical core; escape-hatch tiers for the general case; deterministic disambiguation; and compilation maintained incrementally with periodic recompaction — the same architecture as the borrow checker, the compiler, and the rewriting engine, which is why one set of mechanisms serves them all.

## Identities, names, and metadata

How identities are stored, named, and annotated: the node store, the optional name index, and how lifetimes and permissions attach without bloating every node. *(These are design decisions not yet validated against a real corpus; the bootstrap can use the trivial version of each and tighten later.)*

**The store is keyed by address.** An identity is an interned handle that lives in some backing store — an array, an arena, or any allocator that hands out stable addresses — in which the identity's *address is its id*. Because every node is a fixed two-pointer (16-byte) cell, that address is literally an index; the store needs no separate key and no hash: the id *is* the address. A content → id hash is a *separate, optional* index, needed only where structural **sharing** is wanted — interned canonical identities (one `int32` referenced everywhere) and the e-graph. Ordinary source nodes are not deduped: each occurrence of `a + a` is its own node with its own address. Identities are *per-run*: addresses, and hence ids, are assigned at runtime and are never persisted. The durable form of any graph is source text — files are canonical, promotion serializes by unparsing — and the graph is rebuilt at load, with disposable content-keyed caches as accelerators where startup cost warrants. Anything that must survive a session (caches, compiled artifacts, certificates) keys by content or structure, never by id.

**Names are an optional index, not the management mechanism.** Every identity — named or not — is reached by its handle and by structural traversal from its uses, so the handle is what you compare, store, and refer to. A name is a second index (string → handle) for the subset that source refers to by word — operators, types, declarations, keywords. Most identities are anonymous subexpressions and carry none; naming them all would mean gensym-ing millions of meaningless names, which is bloat, not management.

**Metadata has three homes, chosen by frequency and origin; the model is uniform, the storage is not.** *(Revised June 2026; supersedes the earlier wrapping design.)* A wrapper around a node would make every read of its type yield the wrapper, and a wrapper spliced into the value chain would falsify the prime invariant that the type defines how the value is read — so no facet wraps. Instead a facet lives where it is naturally read: **(1) universal and tiny** — the construction phase, the four bits every node passes through — rides in the node's own tag bits (see *Mutability and construction*), read on every access because the node is already in hand. **(2) Syntactic deviations** — what the user writes on a declaration: `pub`, per-field `mut`, capability and backend modifiers — are read off the declaration's structure at resolution time, the same lookup `.` already performs, so they are not side-data at all. **(3) Inferred deviations** — facts analysis derives that are *not* in the source: a lifetime extended past its scope by escape, a borrow state — are reflectable nodes reached through the derived id → metadata index below. Intrinsic inferred facts (a non-default lifetime) share one per-node metadata record found by that index; relational and event facts are independently-reachable nodes by nature — a borrow references its target, a move relocates a scope-level drop node — so they need no index entry on the target. The giant common case deviates in none of the three: scope-tree defaults supply lifetime (the enclosing scope) and visibility (own scope and descendants) with nothing stored, the phase bits clear at freeze, and the node stays sixteen bytes. The asymmetry is the rule — *a deviation the user writes lives in the declaration; a deviation analysis infers lives in a fact-node* — and a facet that can be either, such as an annotated versus an inferred lifetime, enters by whichever door it came from. (Source positions are not a node facet at all: they live in the persistent token record, which is one-to-one with source and absent for runtime-generated nodes; fine-grained permissions are carried by the reference, per *Reference-granular permissions*.)

**Lookup rides the scoped walk; the index is only an accelerator.** During checking, lexical scoping guarantees a binding's declaration is an *ancestor* of every use, so a top-down walk carrying its environment (push bindings on scope entry, pop on exit — standard scoped-checker behavior) always has the binding's metadata in hand at a use; no index is needed for the batch pass. Cross-references that skip ancestors — to types, functions, definitions elsewhere — are exactly the `'static` / default cases with no non-trivial lifetime to look up. A derived index (id → metadata node) earns its keep only for out-of-walk access: re-checking a single node after an edit, or a tooling hover, where there is no environment built up.

**A uniform model is not uniform storage.** Every identity *has* a permission and a lifetime in the model, but almost none *store* one. The node stays lean — just its type and value (16 bytes), the handle being its address rather than a stored field; metadata hangs off only on deviation; defaults are inherited from the scope tree (permissions on scopes, lifetime from the enclosing scope, dropped at its end); and the giant common case — immutable `'static` definitions: operators, types, the standard library — shares one default of readable-by-all, with no lifetime tracking, no drop node, no borrow state. Storage is proportional to *deviations*, not to identity count, which is what makes a uniform model affordable.

## Standard library and ecosystem strategy

The standard library is built **before** release, avoiding the fragmentation that plagues languages shipping thin stdlibs. By the time Logos is available, the standard ways to do common things are decided, documented, idiomatic, and tested by real programs.

It includes: collections and the iterator protocol `for` desugars onto; Unicode-aware strings and parsing; numerics including arbitrary precision and the cost functions for stability-aware rewriting; symbolic math (expressions, algebraic rule sets, differentiation, integration, polynomial and linear algebra, domain support, verified rules where possible); allocators and smart pointers; concurrency and async primitives; I/O, networking, and filesystem (files and folders as Logic Graph values); verification utilities (SMT integration, refinement predicates, inductive types, common tactics); the compiler-extension framework; and the documentation generator.

The cultural commitment matches the technical one: **single canonical solutions, opinions formed, batteries included, idioms documented.** Competing libraries exist, but the standard library defines what idiomatic Logos looks like, and the language's design is opinionated enough to back it.

## Tooling philosophy

- **LSP-first.** A Logos-written LSP server delivers highlighting, errors, autocomplete, go-to-definition, hover, and refactoring to any LSP editor, so users keep the editors they already have. Because so much information is already in the Logic Graph, the server is richer than equivalents in other languages.
- **Standard-protocol debugging and profiling.** DAP, standard samplers, flamegraphs — users keep the tools they know.
- **A Logos-native structural editor and IDE as long-term work.** Operating directly on Logic Graphs: semantic rather than textual editing, principled rather than heuristic refactoring, the editor itself customizable in Logos and able to ask "update all related occurrences?" with confidence. The effort is real but far smaller than equivalent IDEs elsewhere, because the meta-reflection does the heavy lifting that would otherwise require separate AST, analyzer, refactoring, and language-server stacks. Not on the critical path — LSP covers the immediate need.
- **Documentation generation as a flagship feature.** With types, signatures, doc comments, examples, capability declarations, and proofs all in the graph, generated docs are richer than conventional tools — verified examples, proof obligations, capability summaries — and the generator is customizable Logos.

## Feasibility and effort

The accounting of relative effort and risk:

- **Bootstrap seed (Rust)** — bounded; the hard part is restraint, keeping it small enough to audit.
- **Base language (self-hosted)** — the bulk of the language work, but well-understood territory (borrow checking, inference, data structures) and strongly accelerated by LLM-assisted implementation.
- **Rewriting engine** — bounded; the algorithm is established (egg-style equality saturation) and adaptable to the Logic Graph. Cost-function infrastructure and rule authoring take longer to get right.
- **Symbolic math stdlib** — the most uncertain. Differentiation, basic integration, polynomial arithmetic, and simplification are achievable early; Mathematica-level coverage is a long mathematical-engineering effort. Logos need not reach that bar to be valuable — an 80%-case library with compile-time symbolic evaluation is more useful than another runtime CAS clone.
- **Refinement types + SMT** — established design space (Liquid Haskell, F\*, Dafny); integrating SMT discharge with the base type system and building tooling is real but bounded.
- **Full dependent types + proof terms** — the deepest, longest piece, building on the same foundations Lean matured over years. Not needed for an initial release; the foundation can be architecturally present while the implementation accretes.
- **Backends** — Cranelift and LLVM are well-documented with many existing integrations to learn from; custom backends (embedded, WASM, GPU) are smaller projects given a good interface.
- **Standard library** — ongoing for the language's lifetime; the pre-release subset must cover collections, numerics, I/O, concurrency, basic symbolic math, and verification utilities.
- **LSP server** — less than building from scratch elsewhere because the information is already in the graph.
- **Documentation generator** — modest; the graph carries everything, the work is rendering and cross-references.
- **Structural editor / IDE** — large, but far smaller than equivalent IDEs elsewhere; not on the critical path.
- **Debugger/profiler integration** — standard-protocol integration first; Logos-native versions later.
- **Compiler-extension framework** — partly free from meta-reflection; conventions and tooling mature alongside the standard library, which demonstrates the patterns.
- **Verified demonstration projects** — a verified kernel module, crypto library, or compiler pass; research-grade, and what establishes credibility in the verification niche.

**Accelerants Logos has that earlier language projects lacked:** LLM-assisted implementation, mature compiler infrastructure (Cranelift, LLVM, egg), well-understood foundations (dependent type theory, SMT, borrow checking), and a decade of accumulated PL research. None make the project trivial; each meaningfully reduces effort versus inventing each piece.

**Decelerants:** the unification ambition is genuinely novel in the systems-language space, so some design problems surface through implementation rather than prediction. The Logic-Graph-contains-everything commitment is tested by every feature, and keeping it coherent demands sustained architectural discipline. The verification stratum is where unexpected issues are most likely.

## Closing

The vision is large, and honest about being large. Each piece has working precedents elsewhere — Lean 4's self-hosting, MLIR's layered IR, egg's equality saturation, Rust's borrow checking, Smalltalk's malleable system — and the contribution is unifying them rather than inventing from scratch. The risks are real but the architecture is sound and coherent.
